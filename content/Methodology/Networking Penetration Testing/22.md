## Overview

::card
---
icon: i-lucide-info
title: What is Port 22?
---
**SSH** (Secure Shell) operates on **TCP port 22** as the de facto standard for **secure remote administration**, file transfer (SCP/SFTP), and tunneling. SSH provides **encrypted communication**, strong authentication (password, public key, certificates), and integrity checking — replacing insecure predecessors like Telnet and rlogin. SSH is present on **virtually every Linux/Unix server, network device, cloud instance, and IoT device**. Despite its encryption, SSH remains a prime target — weak credentials, key mismanagement, misconfiguration, and implementation vulnerabilities make it one of the **most attacked services on the internet**.
::

::callout
---
icon: i-lucide-skull
color: red
---
SSH is the **gateway to the kingdom**. Compromising SSH credentials or keys gives an attacker **direct interactive shell access** to the target system — often with elevated privileges. It is the **#1 initial access vector** for Linux/Unix system compromise and the **primary persistence mechanism** for attackers post-exploitation.
::

## Enumeration

### Nmap NSE Scripts

::note
Nmap provides multiple SSH-specific NSE scripts for algorithm enumeration, key extraction, brute forcing, and vulnerability detection.
::

::code-group
```bash [All SSH Scripts]
# Run ALL SSH scripts
nmap -p 22 --script="ssh-*" <TARGET_IP>

# Safe enumeration combo
nmap -p 22 --script="ssh2-enum-algos,ssh-hostkey,ssh-auth-methods,\
sshv1" <TARGET_IP>
```

```bash [Algorithm Enumeration]
# Enumerate supported algorithms
nmap -p 22 --script=ssh2-enum-algos <TARGET_IP>

# Get host keys
nmap -p 22 --script=ssh-hostkey <TARGET_IP>

# Get host keys with full output
nmap -p 22 --script=ssh-hostkey \
  --script-args ssh_hostkey=full <TARGET_IP>

# Check for SSHv1 (should not exist)
nmap -p 22 --script=sshv1 <TARGET_IP>

# Check authentication methods
nmap -p 22 --script=ssh-auth-methods <TARGET_IP>

# Check auth methods for specific user
nmap -p 22 --script=ssh-auth-methods \
  --script-args="ssh.user=root" <TARGET_IP>
```

```bash [Brute Force]
# SSH brute force
nmap -p 22 --script=ssh-brute \
  --script-args userdb=users.txt,passdb=passwords.txt <TARGET_IP>

# With specific timing
nmap -p 22 --script=ssh-brute \
  --script-args "userdb=users.txt,passdb=passwords.txt,\
ssh-brute.timeout=5s" <TARGET_IP>
```

```bash [Vulnerability Checks]
# SSH public key acceptance check
nmap -p 22 --script=ssh-publickey-acceptance \
  --script-args="ssh.publickey=/path/to/key.pub" <TARGET_IP>

# Run script for known SSH vulnerabilities
nmap -p 22 --script="ssh-*,sshv1" <TARGET_IP>
```

### Manual Enumeration

::steps{level="4"}

#### Identify Authentication Methods

```bash [Terminal]
# Method 1: SSH verbose connection
ssh -v user@<TARGET_IP> 2>&1 | grep "Authentications that can continue"
# Output: Authentications that can continue: publickey,password,keyboard-interactive

# Method 2: Force none auth (reveals available methods)
ssh -o PreferredAuthentications=none \
  -o PasswordAuthentication=no \
  nonexistent@<TARGET_IP> 2>&1
# Output: Permission denied (publickey,password,keyboard-interactive).

# Method 3: Nmap
nmap -p 22 --script=ssh-auth-methods \
  --script-args="ssh.user=root" <TARGET_IP>

# Method 4: Check multiple users
for user in root admin administrator user test guest deploy \
  git jenkins ansible vagrant ubuntu ec2-user centos; do
  echo -n "$user: "
  ssh -o PreferredAuthentications=none \
    -o PasswordAuthentication=no \
    -o ConnectTimeout=3 \
    $user@<TARGET_IP> 2>&1 | grep "Permission denied" | \
    sed 's/.*(\(.*\))/\1/'
done
```

## Weakness Analysis

### Core Vulnerabilities

::card-group
  ::card
  ---
  icon: i-lucide-key
  title: "1. Weak / Default Credentials"
  color: red
  ---
  The **#1 SSH attack vector**. Blank passwords, default vendor credentials, weak user passwords, and credential reuse. Botnets like Mirai/RapperBot scan billions of IPs for weak SSH passwords.
  ::

  ::card
  ---
  icon: i-lucide-file-key
  title: "2. Leaked / Stolen SSH Keys"
  color: red
  ---
  Private keys exposed in **Git repositories, Docker images, backups, CI/CD pipelines, and paste sites**. A single leaked `id_rsa` can compromise entire infrastructure.
  ::

  ::card
  ---
  icon: i-lucide-copy
  title: "3. Shared / Duplicate Host Keys"
  color: red
  ---
  Cloned VMs, containers, and cloud images share **identical host keys**. Allows impersonation and MITM attacks. Golden images must regenerate keys.
  ::

  ::card
  ---
  icon: i-lucide-shield-off
  title: "4. Weak Algorithms"
  color: orange
  ---
  Support for **SSHv1, DSA keys, SHA-1 signatures, CBC ciphers, RC4/arcfour, weak DH groups**. Enable downgrade attacks and cryptographic weaknesses.
  ::

  ::card
  ---
  icon: i-lucide-settings
  title: "5. Misconfiguration"
  color: orange
  ---
  **Root login enabled, password auth with no rate limiting, no fail2ban, permissive authorized_keys, agent forwarding exposed, X11 forwarding enabled unnecessarily.**
  ::

  ::card
  ---
  icon: i-lucide-arrow-right-left
  title: "6. SSH Tunneling / Pivoting"
  color: orange
  ---
  Compromised SSH access enables **port forwarding, SOCKS proxy, and tunneling** — allowing attackers to pivot into internal networks through a single SSH session.
  ::
::

### Detailed Attack Surface

::field-group
  ::field{name="Password Brute Force" type="Critical"}
  Automated credential stuffing and dictionary attacks against SSH. Tools like Hydra, Medusa, and Patator can test thousands of credentials. Without rate limiting or fail2ban, success is highly likely.
  ::

  ::field{name="Private Key Theft" type="Critical"}
  SSH private keys leaked via Git repos, backups, NFS shares, Docker layers, CI/CD variables, or post-exploitation file collection. No password needed if key is unencrypted.
  ::

  ::field{name="Key Passphrase Cracking" type="High"}
  Encrypted SSH private keys can be cracked offline with `ssh2john` + Hashcat/John. Weak passphrases fall quickly to dictionary attacks.
  ::

  ::field{name="SSH Agent Hijacking" type="High"}
  If `SSH_AUTH_SOCK` is accessible (e.g., via shared `/tmp`), an attacker can use a victim's forwarded SSH agent to authenticate to other systems **as the victim**.
  ::

  ::field{name="SSH Agent Forwarding Abuse" type="High"}
  With `ForwardAgent yes`, a compromised intermediate server can use the user's SSH agent to connect to further servers — escalating lateral movement.
  ::

  ::field{name="Username Enumeration" type="Medium"}
  CVE-2016-6210, CVE-2018-15473. Timing differences or error responses reveal valid usernames, enabling targeted brute-force attacks.
  ::

  ::field{name="Known Hosts Harvesting" type="Medium"}
  Reading `~/.ssh/known_hosts` reveals all servers a user connects to — mapping internal infrastructure and identifying lateral movement targets.
  ::

  ::field{name="Authorized Keys Injection" type="Critical"}
  Write attacker's public key to `~/.ssh/authorized_keys` for persistent, password-less access that survives password changes.
  ::

  ::field{name="SSH Tunneling / SOCKS Proxy" type="High"}
  Compromised SSH creates encrypted tunnels into the internal network. Dynamic SOCKS proxy allows the attacker to route **all traffic** through the target.
  ::

  ::field{name="Port Forwarding Abuse" type="High"}
  SSH local/remote port forwarding exposes internal services (databases, admin panels) to the attacker's machine through encrypted tunnels.
  ::

  ::field{name="MITM via Host Key" type="High"}

### Phase 5 — SSH Tunneling & Pivoting

::warning
Once SSH access is obtained, **tunneling and pivoting** transform a single compromised host into a gateway to the entire internal network.
::

::tabs
  :::tabs-item{icon="i-lucide-arrow-right" label="Local Port Forwarding"}
  ```bash [Terminal]
  # ─── CONCEPT ───
  # Forward a port from the TARGET's internal network to your machine
  # Syntax: ssh -L <local_port>:<internal_host>:<internal_port> user@<TARGET>

  # Access internal web server (not directly reachable)
  ssh -L 8080:10.0.0.5:80 user@<TARGET_IP>
  # Now browse: http://127.0.0.1:8080 → reaches 10.0.0.5:80

  # Access internal database
  ssh -L 3306:10.0.0.10:3306 user@<TARGET_IP>
  mysql -h 127.0.0.1 -P 3306 -u root -p

  # Access internal RDP
  ssh -L 3389:10.0.0.20:3389 user@<TARGET_IP>
  xfreerdp /v:127.0.0.1 /u:admin /p:password

  # Multiple forwards
  ssh -L 8080:10.0.0.5:80 \
      -L 3306:10.0.0.10:3306 \
      -L 445:10.0.0.15:445 \
      user@<TARGET_IP>

  # Background tunnel
  ssh -L 8080:10.0.0.5:80 -fN user@<TARGET_IP>
  ```
  :::

  :::tabs-item{icon="i-lucide-arrow-left" label="Remote Port Forwarding"}
  ```bash [Terminal]
  # ─── CONCEPT ───
  # Expose your local service through the TARGET
  # Useful for callbacks when target can't reach you directly
  # Syntax: ssh -R <remote_port>:<local_host>:<local_port> user@<TARGET>

  # Expose your web server through target
  ssh -R 8888:127.0.0.1:80 user@<TARGET_IP>
  # Now <TARGET_IP>:8888 reaches your local port 80

  # Expose your listener for reverse shells
  ssh -R 4444:127.0.0.1:4444 user@<TARGET_IP>
  # Internal machines connect to <TARGET_IP>:4444
  # → forwarded to your nc listener on port 4444

  # Background
  ssh -R 8888:127.0.0.1:80 -fN user@<TARGET_IP>

  # NOTE: requires GatewayPorts yes in target's sshd_config
  # for binding to 0.0.0.0 instead of 127.0.0.1
  ```
  :::

  :::tabs-item{icon="i-lucide-globe" label="Dynamic SOCKS Proxy"}
  ```bash [Terminal]
  # ─── CONCEPT ───
  # Create a SOCKS proxy through SSH → route ALL traffic through target
  # Syntax: ssh -D <local_port> user@<TARGET>

  # Create SOCKS5 proxy
  ssh -D 1080 user@<TARGET_IP>
  ssh -D 9050 -fN user@<TARGET_IP>  # Background

  # ─── USE WITH PROXYCHAINS ───
  # Edit /etc/proxychains4.conf:
  # socks5 127.0.0.1 1080

  # Run tools through proxy
  proxychains4 nmap -sT -p 80,443,22,3306 10.0.0.0/24
  proxychains4 curl http://10.0.0.5/admin
  proxychains4 firefox
  proxychains4 crackmapexec smb 10.0.0.0/24

  # ─── USE WITH BROWSER ───
  # Firefox: Settings → Network → Manual Proxy
  # SOCKS Host: 127.0.0.1, Port: 1080, SOCKS v5
  # Check: "Proxy DNS when using SOCKS v5"

  # ─── USE WITH BURP SUITE ───
  # User Options → SOCKS Proxy
  # Host: 127.0.0.1, Port: 1080
  ```
  :::

  :::tabs-item{icon="i-lucide-layers" label="Multi-Hop / ProxyJump"}
  ```bash [Terminal]
  # ─── PROXYJUMP (Modern SSH) ───
  # Jump through intermediary to reach internal host
  ssh -J user@jumpbox:22 admin@internal-server

  # Multiple jumps
  ssh -J user@jump1,user@jump2 admin@final-target

  # ─── SSH CONFIG (Persistent Tunnels) ───
  cat >> ~/.ssh/config << 'EOF'
  
  Host jumpbox
    HostName <JUMP_IP>
    User user
    IdentityFile ~/.ssh/id_rsa
  
  Host internal-*
    ProxyJump jumpbox
    User admin
  
  Host internal-db
    HostName 10.0.0.10
  
  Host internal-web
    HostName 10.0.0.5
  
  EOF

  # Now simply:
  ssh internal-db
  ssh internal-web

  # ─── SSHUTTLE (VPN over SSH) ───
  # Route entire subnet through SSH
  sshuttle -r user@<TARGET_IP> 10.0.0.0/24
  sshuttle -r user@<TARGET_IP> 10.0.0.0/24 172.16.0.0/16

  # With specific DNS
  sshuttle --dns -r user@<TARGET_IP> 10.0.0.0/24

  # Exclude certain IPs
  sshuttle -r user@<TARGET_IP> 10.0.0.0/24 -x 10.0.0.1
  ```

## Defense & Remediation

::tip
Provide these **remediation recommendations** in your pentest report for SSH findings.
::

::card-group
  ::card
  ---
  icon: i-lucide-key-round
  title: "Key-Based Auth Only"
  color: green
  ---
  Disable `PasswordAuthentication`. Use **Ed25519 or RSA 4096-bit keys** exclusively. Require key passphrases. Disable `ChallengeResponseAuthentication`.
  ::

  ::card
  ---
  icon: i-lucide-shield-check
  title: "Disable Root Login"
  color: green
  ---
  Set `PermitRootLogin no` (or `prohibit-password` minimum). Require users to `sudo` for privilege escalation with proper auditing.
  ::

  ::card
  ---
  icon: i-lucide-shield-ban
  title: "Implement fail2ban"
  color: green
  ---
  Install and configure **fail2ban** to block IPs after failed login attempts. Default: ban after 5 failures for 10 minutes. Consider permanent bans for repeat offenders.
  ::

  ::card
  ---
  icon: i-lucide-lock
  title: "Strong Algorithms Only"
  color: green
  ---
  Disable **SSHv1, DSA, SHA-1 KEX, CBC ciphers, RC4, weak DH groups**. Use only `curve25519-sha256`, `aes256-gcm`, `hmac-sha2-256-etm`, `ssh-ed25519`.
  ::
::

::card-group
  ::card
  ---
  icon: i-lucide-network
  title: "Network Restrictions"
  color: green
  ---
  Restrict SSH access via **firewall rules** to specific IPs/VPNs. Use **port knocking** or **SPA (Single Packet Authorization)** like `fwknop`. Move SSH to a non-standard port as defense-in-depth.
  ::

  ::card
  ---
  icon: i-lucide-users
  title: "Multi-Factor Authentication"
  color: green
  ---
  Implement **MFA via Google Authenticator PAM, Duo Security, or YubiKey**. Require both SSH key + OTP for critical systems.
  ::

  ::card
  ---
  icon: i-lucide-key
  title: "Key Management"
  color: green
  ---
  **Audit authorized_keys** regularly. Remove unused keys. Rotate host keys after VM cloning. Use **SSH certificates** instead of raw authorized_keys for enterprise scale.
  ::

  ::card
  ---
  icon: i-lucide-shield-off
  title: "Disable Forwarding"
  color: green
  ---
  Set `AllowAgentForwarding no`, `AllowTcpForwarding no`, `X11Forwarding no`, `PermitTunnel no` unless specifically required. Use `ProxyJump` instead of agent forwarding.
  ::

  ::card
  ---
  icon: i-lucide-refresh-cw
  title: "Patch & Update"
  color: green
  ---
  Keep OpenSSH **updated** to the latest version. Patch **immediately** for critical CVEs like regreSSHion. Subscribe to OpenSSH security advisories.
  ::

  ::card
  ---
  icon: i-lucide-eye
  title: "Monitoring & Logging"
  color: green
  ---
  Enable **verbose SSH logging** (`LogLevel VERBOSE`). Monitor auth.log for brute force patterns. Alert on new authorized_keys additions. Forward logs to SIEM.
  ::
